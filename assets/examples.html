Sequence / BASICS / Simple Message

這個範例告訴你「角色 A 對角色 B 說一句話」。就像兩個人互相傳紙條。

為什麼要用

  • 把系統裡的對話順序畫出來，老師或同學一看就懂。
  • 適合講「誰先說、誰再回」的故事。

怎麼用

  1. 先寫 @startuml 和 @enduml。
  2. 用 Alice -> Bob: 訊息 表示 A 對 B 說話。

可能的用法

  • 登入流程：使用者 → 系統 → 伺服器。
  • 聊天機器人：你問 → 機器人回。

修改練習（12 歲也懂）

  1. 練習一：把 Alice、Bob 改成你喜歡的角色名字，再把訊息改成一句日常對話。
  2. 練習二：先新增一行 participant 宣告角色，再把原本的訊息改成「先問、再回」的內容。
  3. 練習三：把訊息改成兩行，第一行問問題，第二行請對方回答。
  4. 練習四：把對話順序換一下（誰先說換成另一個人），再改訊息內容。
  5. 練習五：加一個第三個角色，讓他也說一句話。

PlantUML 語法規則（完整清單）

  • 以 @startuml 開頭、@enduml 結尾。
  • 角色可以寫 participant 或直接用名字。
  • A -> B 是同步訊息，A --> B 是回傳。
  • 文字寫在冒號後面：A -> B: Hello。
  • 順序由上到下表示時間。
  • 可以連續多行訊息，表示多段對話。
  • 訊息文字可省略：A -> B 也合法。
  • 角色名稱可包含空白，需用引號包起來。

Sequence / BASICS / Return Message

這個範例示範「請求與回應」。像你問問題、對方回答。

為什麼要用

  • 清楚表達「發問」和「回覆」兩個步驟。

怎麼用

  1. A -> B 表示請求。
  2. B --> A 表示回應。

可能的用法

  • API 呼叫：前端 → 後端 → 回傳資料。

修改練習（12 歲也懂）

  1. 練習一：把角色改成 Client 與 Server，再把訊息改成「要資料 / 傳資料」。
  2. 練習二：先把請求訊息改成「查詢」，再把回覆訊息改成「查詢結果」。
  3. 練習三：加一個新的請求和回覆（共四行），表示連續問兩次。
  4. 練習四：把回覆改成另一個人回，並調整訊息文字。
  5. 練習五：在請求前加一行「準備中」的自我訊息。

PlantUML 語法規則（完整清單）

  • -> 用於發出訊息。
  • --> 常用於回傳訊息。
  • 可加上文字描述回傳內容。
  • 請求與回應通常成對出現，但不是硬性規定。
  • 可在訊息前先宣告角色，讓名稱更清楚。
  • 訊息順序由上到下代表時間先後。

Sequence / BASICS / Self-Message

這是「自己跟自己說話」，代表自己在做內部步驟。

為什麼要用

  • 顯示物件自己進行處理，不需要別人幫忙。

怎麼用

  1. Alice -> Alice: Process

可能的用法

  • 演算法步驟、內部計算。

修改練習（12 歲也懂）

  1. 練習一：先把 Alice 改成你喜歡的名字（例如 Me），再把訊息改成你今天想做的事。
  2. 練習二：先把第一行的動作改成 先讀資料，再加一行 Me -> Me: 再檢查。
  3. 練習三：先把訊息改成 想一想，再加一行 Me -> Me: 記下重點。
  4. 練習四：先把名字改成 Robot，再加一行 Robot -> Robot: 充電。
  5. 練習五：先新增一行 Me -> Me: 開始 放在最上面，再在最後加一行 Me -> Me: 完成。

PlantUML 語法規則（完整清單）

  • 箭頭左右兩邊可以是同一個角色。
  • 基本格式：角色 -> 角色: 文字。
  • 角色名字可直接使用，不一定要先宣告。
  • 訊息文字寫在冒號後面，沒有文字也可以。
  • 同一個角色可連續寫多行，表示多個內部步驟。
  • 可搭配 activate / deactivate 顯示忙碌時間。
  • 順序由上到下表示時間先後。

Sequence / PARTICIPANTS / Types

這個範例讓不同角色用不同圖示出現，像「人、系統、資料庫」。

為什麼要用

  • 用圖示區分角色，讀圖更快。

怎麼用

  1. 用 actor、boundary、control、database、entity。

可能的用法

  • 登入流程：使用者 → 網頁 → 邏輯 → 資料庫。

修改練習（12 歲也懂）

  1. 練習一：把 User 改成 Student，再把訊息改成「登入作業」。
  2. 練習二：先加一個 boundary App，再把箭頭改成先到 App。
  3. 練習三：把 database DB 改成 database Books，再改訊息內容。
  4. 練習四：新增一個 control 角色，並加一條互動訊息。
  5. 練習五：把角色順序調整，再讓訊息也跟著改順序。

PlantUML 語法規則（完整清單）

  • 角色類型會改變外觀但名字不變。
  • 同樣可以使用 -> 畫互動。
  • 角色宣告格式：actor/ boundary/ control/ database/ entity 名稱。
  • 角色可以先宣告，再在訊息中直接使用名字。
  • 未宣告的名字會被當作一般角色。

Sequence / PARTICIPANTS / Colors & Styles

這個範例示範「上色與樣式」。讓圖更漂亮、重點更清楚。

為什麼要用

  • 用顏色標示重要角色。

怎麼用

  1. 用 skinparam 調整樣式。
  2. 在角色後面加色碼：participant Alice #AliceBlue。

可能的用法

  • 把「核心系統」用亮色表示。

修改練習（12 歲也懂）

  1. 練習一：把 Alice 改成你喜歡的名字，再換成另一種顏色。
  2. 練習二：先把訊息改成「開始」，再把另一個角色改成深色。
  3. 練習三：新增第三個角色，並替它指定不同顏色。
  4. 練習四：調整 skinparam sequenceMessageAlign 的值，再換一條訊息。
  5. 練習五：把一位角色換成樣式名稱（如 #technology），再改訊息內容。

PlantUML 語法規則（完整清單）

  • skinparam 可調整對齊、字型、顏色等。
  • 角色後可接顏色或樣式名稱。
  • 顏色可用色名或色碼，例如 #AliceBlue。
  • 同一角色若重複宣告，以最後一次為準。
  • 樣式設定不影響訊息語法。

Sequence / CONTROL FLOW / Alt/Else (Choices)

這個範例表示「如果…不然…」。像選擇題。

為什麼要用

  • 畫出不同情況下的分支行為。

怎麼用

  1. 用 alt 開始，else 切換分支，end 結束。

可能的用法

  • 付款成功或失敗的流程。

修改練習（12 歲也懂）

  1. 練習一：把條件改成「有錢 / 沒錢」，再改兩邊的訊息文字。
  2. 練習二：先新增第三個 else 分支，再補一行訊息。
  3. 練習三：把 alt 的標題改成你自己的情境，再改訊息內容。
  4. 練習四：在每個分支各加一行訊息，讓步驟更清楚。
  5. 練習五：把其中一個分支改成「失敗後再試一次」的對話。

PlantUML 語法規則（完整清單）

  • alt 條件 開始分支。
  • 可有多個 else。
  • 最後以 end 結束。
  • 每個分支內都可以放多行訊息。
  • 條件文字可省略，但建議寫清楚。

Sequence / CONTROL FLOW / Loop

這個範例表示「重複做事情」。

為什麼要用

  • 展示多次重複的動作。

怎麼用

  1. 用 loop 與 end 包住重複內容。

可能的用法

  • 每秒檢查一次狀態。

修改練習（12 歲也懂）

  1. 練習一：把 loop 的文字改成「每 3 次」，再改訊息內容。
  2. 練習二：在迴圈內加一行新訊息，表示兩個動作。
  3. 練習三：把角色名稱改掉，再把訊息改成「檢查中」。
  4. 練習四：把迴圈標題改成條件式，例如「只要還沒完成」。
  5. 練習五：在迴圈外加一行開始與結束的訊息。

PlantUML 語法規則（完整清單）

  • loop 次數/條件 可寫在同一行。
  • end 結束迴圈區塊。
  • 迴圈內可有多行訊息。

Sequence / CONTROL FLOW / Parallel (Par)

這個範例表示「同時做兩件事」。

為什麼要用

  • 顯示並行工作。

怎麼用

  1. 用 par 開始，else 切換並行分支，end 結束。

可能的用法

  • 同時通知兩個服務。

修改練習（12 歲也懂）

  1. 練習一：把兩個分支的訊息改成「發信」與「傳簡訊」。
  2. 練習二：先把角色改名，再把每個分支多加一行訊息。
  3. 練習三：新增第三個並行分支（再加一個 else），並寫一行訊息。
  4. 練習四：把其中一個分支改成「慢慢做」，並加一行結束訊息。
  5. 練習五：在 par 前加一行「開始任務」訊息，在 end 後加一行「完成」。

PlantUML 語法規則（完整清單）

  • par 代表並行區塊。
  • 每個分支可有自己的訊息序列。
  • else 用來切換到另一個並行分支。
  • 最後以 end 結束並行區塊。

Sequence / ADVANCED / Activation

這個範例用「激活條」表示某個角色正在忙碌。

為什麼要用

  • 表達誰在執行工作，以及工作持續多久。

怎麼用

  1. 用 activate 角色 開始。
  2. 用 deactivate 角色 結束。

可能的用法

  • 顯示系統正在處理請求。

修改練習（12 歲也懂）

  1. 練習一：把角色改成 Server，再把訊息改成「處理中」。
  2. 練習二：先加一條請求訊息，再把 activate 放在請求後。
  3. 練習三：在忙碌期間加一行訊息，表示「完成一半」。
  4. 練習四：新增另一個角色，讓它也有一段忙碌時間。
  5. 練習五：把忙碌開始與結束的位置往下移一行，觀察差異。

PlantUML 語法規則（完整清單）

  • 可在訊息前後加入 activate/deactivate。
  • activate 角色 與 deactivate 角色 要成對。
  • 可在忙碌期間插入多行訊息。

  • 說明規則或限制。

Use Case / BASICS / Simple Use Case

用例圖是在講「使用者要做什麼功能」。

為什麼要用

  • 快速列出系統有哪些功能。

怎麼用

  1. 用 actor 表示角色。
  2. 用 usecase 表示功能。
  3. 用連線把角色和功能連起來。

可能的用法

  • 設計點餐系統、登入系統。

修改練習（12 歲也懂）

  1. 練習一：把 User 改成 Student，再把功能改成「查成績」。
  2. 練習二：先新增一個功能（另一個 usecase），再把角色連到它。
  3. 練習三：把圖改成由左到右，再把功能名稱改成更短的字。
  4. 練習四：新增第二個角色，並連到同一個功能。
  5. 練習五：替功能加一個別名，再用別名畫連線。

PlantUML 語法規則（完整清單）

  • usecase "名稱" as ID 定義功能。
  • 角色到功能可用 --> 連線。
  • left to right direction 控制方向。
  • actor 用來定義角色外觀。
  • 功能名稱可用引號包住。
  • 連線也可用 -- 或 ..。

Use Case / RELATIONSHIPS / Include/Extend

這個範例示範「功能之間的關係」。

為什麼要用

  • 表達功能共用或加強的關係。

怎麼用

  1. ..> 表示關聯。
  2. 標註 <<include>> 或 <<extend>>。

可能的用法

  • 結帳包含驗證信用卡。

修改練習（12 歲也懂）

  1. 練習一：把功能改成「買票」與「驗票」，再保留 <<include>>。
  2. 練習二：新增一個 <<extend>> 功能，例如「優惠券」。
  3. 練習三：先把箭頭方向換一邊，再調整功能名稱。
  4. 練習四：把 include 改成 extend，並修改說明文字。
  5. 練習五：加一個新角色，並連到主要功能。

PlantUML 語法規則（完整清單）

  • <<include>> 表示一定會用到。
  • <<extend>> 表示可選的擴充。
  • 關係線通常用虛線：..>。
  • 標籤寫在冒號後：..> : <<include>>。

怎麼用

  1. + 公開、- 私有、# 保護、~ 套件。

Class / VISIBILITY / Abstract/Static

這個範例示範抽象與靜態成員。

為什麼要用

  • 說明不需要實例化或必須被覆寫的功能。

怎麼用

  1. 用 {static} 或 {abstract}。

可能的用法

  • 工具類別、共同規格。

修改練習（12 歲也懂）
