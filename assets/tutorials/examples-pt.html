Sequence / BASICS / Simple Message

Este exemplo mostra “A pessoa A envia uma mensagem para a pessoa B”, como passar um bilhete.

Por que usar

  • Mostra a ordem da conversa de forma fácil de entender.
  • Ótimo para histórias de “quem fala primeiro, quem responde depois”.

Como usar

  1. Escreva @startuml e @enduml.
  2. Use Alice -> Bob: Message para mostrar A falando com B.

Possíveis casos de uso

  • Fluxo de login: Usuário → Sistema → Servidor.
  • Chatbot: Você pergunta → Bot responde.

Prática (fácil para crianças)

  1. Prática 1: Troque Alice e Bob por seus nomes favoritos e mude a mensagem para uma conversa do dia a dia.
  2. Prática 2: Adicione uma linha participant primeiro, depois mude a mensagem para “perguntar primeiro, responder depois”.
  3. Prática 3: Transforme a mensagem em duas linhas: primeiro pergunte, depois responda, e encurte uma linha.
  4. Prática 4: Troque quem fala primeiro e altere o texto da mensagem.
  5. Prática 5: Adicione uma terceira pessoa que também fale uma linha e renomeie uma pessoa.

Regras de sintaxe do PlantUML (lista completa)

  • Comece com @startuml e termine com @enduml.
  • Participantes podem ser declarados com participant ou usados diretamente.
  • A -> B é uma chamada, A --> B é um retorno.
  • Coloque o texto após dois pontos: A -> B: Hello.
  • De cima para baixo mostra a ordem do tempo.
  • Várias linhas de mensagem mostram uma conversa maior.
  • O texto da mensagem pode ser omitido: A -> B é válido.
  • Nomes com espaços devem estar entre aspas.

Sequence / BASICS / Return Message

Este exemplo mostra “pedido e resposta”, como pergunta e resposta.

Por que usar

  • Mostra claramente as etapas do pedido e da resposta.

Como usar

  1. A -> B é o pedido.
  2. B --> A é a resposta.

Possíveis casos de uso

  • Chamada de API: Frontend → Backend → Retorna dados.

Prática (fácil para crianças)

  1. Prática 1: Troque os papéis para Client e Server e as mensagens para “obter dados / retornar dados”.
  2. Prática 2: Troque o pedido para “query” e a resposta para “query result”.
  3. Prática 3: Adicione outro pedido e resposta (quatro linhas no total) para mostrar duas perguntas, e mude o segundo texto.
  4. Prática 4: Faça outra pessoa responder e ajuste o texto.
  5. Prática 5: Adicione uma mensagem própria de “preparando” antes do pedido e deixe o pedido mais específico.

Regras de sintaxe do PlantUML (lista completa)

  • -> envia uma mensagem.
  • --> normalmente indica um retorno.
  • Adicione texto para descrever o conteúdo da resposta.
  • Pedidos e respostas costumam vir em pares, mas não é obrigatório.
  • Declare participantes primeiro se quiser clareza.
  • O tempo flui de cima para baixo.

Sequence / BASICS / Self-Message

Isso é “falar consigo mesmo”, mostrando um passo interno.

Por que usar

  • Mostra um objeto fazendo trabalho interno sem outros.

Como usar

  1. Alice -> Alice: Process

Possíveis casos de uso

  • Passos de algoritmo, cálculos internos.

Prática (fácil para crianças)

  1. Prática 1: Troque Alice pelo seu nome (ex.: Me) e mude a mensagem para algo que você quer fazer hoje.
  2. Prática 2: Mude a ação para “ler dados” e adicione Me -> Me: verificar novamente.
  3. Prática 3: Mude a mensagem para “pensar” e adicione Me -> Me: escrever pontos-chave.
  4. Prática 4: Mude o nome para Robot e adicione Robot -> Robot: recarregar.
  5. Prática 5: Adicione Me -> Me: start no topo e Me -> Me: done no fim.

Regras de sintaxe do PlantUML (lista completa)

  • As duas pontas da seta podem ser o mesmo participante.
  • Formato básico: Role -> Role: Text.
  • Nomes podem ser usados sem declaração prévia.
  • Texto após dois pontos é opcional.
  • Várias mensagens próprias mostram vários passos internos.
  • Pode ser usado com activate / deactivate.
  • O tempo flui de cima para baixo.

Sequence / PARTICIPANTS / Types

Este exemplo mostra ícones diferentes para papéis diferentes, como pessoa, sistema e banco de dados.

Por que usar

  • Ícones ajudam a reconhecer os papéis mais rápido.

Como usar

  1. Use actor, boundary, control, database, entity.

Possíveis casos de uso

  • Fluxo de login: User → Web → Logic → Database.

Prática (fácil para crianças)

  1. Prática 1: Renomeie User para Student e mude a mensagem para “login homework”.
  2. Prática 2: Adicione boundary App e direcione a seta primeiro para App.
  3. Prática 3: Renomeie database DB para database Books e mude a mensagem.
  4. Prática 4: Adicione um papel control e uma linha de interação.
  5. Prática 5: Reordene os participantes e ajuste as mensagens.

Regras de sintaxe do PlantUML (lista completa)

  • O tipo de papel muda a aparência, não o nome.
  • Use -> para interações.
  • Formato de declaração: actor/boundary/control/database/entity Name.
  • Declare papéis primeiro e depois use seus nomes.
  • Nomes não declarados são participantes normais.

Sequence / PARTICIPANTS / Colors & Styles

Este exemplo mostra cores e estilos para deixar o diagrama mais claro.

Por que usar

  • Destaque papéis importantes com cor.

Como usar

  1. Use skinparam para ajustar estilos.
  2. Adicione uma cor após um papel: participant Alice #AliceBlue.

Possíveis casos de uso

  • Mostrar o “sistema central” com cor brilhante.

Prática (fácil para crianças)

  1. Prática 1: Renomeie Alice e mude a cor.
  2. Prática 2: Mude a mensagem para “start” e deixe o outro papel mais escuro.
  3. Prática 3: Adicione um terceiro papel e uma cor diferente.
  4. Prática 4: Mude skinparam sequenceMessageAlign e ajuste uma linha.
  5. Prática 5: Use um nome de estilo (como #technology) para um papel e mude a mensagem.

Regras de sintaxe do PlantUML (lista completa)

  • skinparam controla alinhamento, fonte, cores etc.
  • Os papéis podem ter cor ou nome de estilo.
  • As cores podem ser nomes ou hex, como #AliceBlue.
  • Se o mesmo papel for declarado novamente, vale o último.
  • Configurações de estilo não mudam a sintaxe das mensagens.

Sequence / CONTROL FLOW / Alt/Else (Choices)

Este exemplo mostra “se…senão…”, como escolhas múltiplas.

Por que usar

  • Mostrar ramos diferentes para condições diferentes.

Como usar

  1. Use alt para começar, else para outro ramo e end para terminar.

Possíveis casos de uso

  • Pagamento aprovado ou recusado.

Prática (fácil para crianças)

  1. Prática 1: Mude as condições para “tem dinheiro / não tem dinheiro” e altere as mensagens.
  2. Prática 2: Adicione um terceiro ramo else e uma mensagem.
  3. Prática 3: Mude o título do alt para seu cenário e edite as mensagens.
  4. Prática 4: Adicione uma linha em cada ramo e mude os títulos.
  5. Prática 5: Faça um ramo “tentar novamente após falha” e o outro com um final diferente.

Regras de sintaxe do PlantUML (lista completa)

  • alt Condition inicia um ramo.
  • Pode haver vários ramos else.
  • Finalize com end.
  • Cada ramo pode conter várias mensagens.
  • O texto da condição é opcional, mas recomendado.

Como usar

Sequence / CONTROL FLOW / Parallel (Par)

Este exemplo mostra duas coisas acontecendo ao mesmo tempo.

Por que usar

  • Mostrar tarefas paralelas.

Como usar

  1. Use par para começar, else para outro ramo paralelo e end para terminar.

Possíveis casos de uso

  • Notificar dois serviços ao mesmo tempo.

Prática (fácil para crianças)

  1. Prática 1: Mude os ramos para “enviar email” e “enviar SMS” e renomeie os papéis.
  2. Prática 2: Renomeie os papéis e adicione outra linha em cada ramo.
  3. Prática 3: Adicione um terceiro ramo (outro else) e escreva uma mensagem, depois encurte uma linha.
  4. Prática 4: Mude um ramo para “fazer devagar”, adicione uma mensagem de fim e atualize o texto do outro ramo.
  5. Prática 5: Adicione uma mensagem “iniciar tarefa” antes de par e uma mensagem “feito” após end.

Regras de sintaxe do PlantUML (lista completa)

  • par cria um bloco paralelo.
  • Cada ramo pode ter sua própria sequência de mensagens.
  • else troca para outro ramo paralelo.
  • Finalize com end.

Sequence / ADVANCED / Activation

Este exemplo usa barras de ativação para mostrar quando um papel está ocupado.

Por que usar

  • Mostrar quem está trabalhando e por quanto tempo.

Como usar

  1. Use activate Role para começar.
  2. Use deactivate Role para terminar.

Possíveis casos de uso

  • Mostrar o servidor processando um pedido.

Prática (fácil para crianças)

  1. Prática 1: Troque o papel para Server e a mensagem para “processing”.
  2. Prática 2: Adicione uma linha de pedido e coloque activate após o pedido.
  3. Prática 3: Adicione uma mensagem no meio como “halfway” e depois troque pelo seu texto.
  4. Prática 4: Adicione outro papel com sua própria ativação e mensagem.
  5. Prática 5: Mova o início/fim da ativação uma linha para baixo e ajuste uma mensagem.

Regras de sintaxe do PlantUML (lista completa)

  • Você pode colocar activate/deactivate antes ou depois das mensagens.
  • activate e deactivate devem ser usados em par.
  • Mensagens podem ser inseridas enquanto um papel está ativo.
