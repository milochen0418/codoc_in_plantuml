Sequence / BASICS / Simple Message

This example shows “Person A sends one message to Person B,” like passing a note.

Why use it

  • Shows the order of a conversation so it’s easy to understand.
  • Great for “who speaks first, who replies next” stories.

How to use

  1. Write @startuml and @enduml.
  2. Use Alice -> Bob: Message to show A talking to B.

Possible use cases

  • Login flow: User → System → Server.
  • Chatbot: You ask → Bot replies.

Practice (easy for kids)

  1. Practice 1: Change Alice and Bob to your favorite names, and change the message to a daily conversation.
  2. Practice 2: Add a participant line first, then change the message to “ask first, then answer.”
  3. Practice 3: Change the message into two lines: first ask, then answer, and shorten one line.
  4. Practice 4: Swap who speaks first, then change the message text.
  5. Practice 5: Add a third person who also says one line, then rename one person.

PlantUML syntax rules (complete list)

  • Start with @startuml and end with @enduml.
  • Participants can be declared with participant or used directly.
  • A -> B is a call, A --> B is a return.
  • Put text after a colon: A -> B: Hello.
  • Top-to-bottom shows time order.
  • Multiple message lines show a longer conversation.
  • Message text can be omitted: A -> B is valid.
  • Names with spaces should be in quotes.

Sequence / BASICS / Return Message

This example shows “request and response,” like a question and answer.

Why use it

  • Clearly shows the request and the reply steps.

How to use

  1. A -> B is the request.
  2. B --> A is the response.

Possible use cases

  • API call: Frontend → Backend → Return data.

Practice (easy for kids)

  1. Practice 1: Change roles to Client and Server, and messages to “get data / return data.”
  2. Practice 2: Change the request to “query,” and the reply to “query result.”
  3. Practice 3: Add another request and reply (four lines total) to show two questions, and change the second text.
  4. Practice 4: Make a different person reply, and adjust the text.
  5. Practice 5: Add a “preparing” self-message before the request, then make the request more specific.

PlantUML syntax rules (complete list)

  • -> sends a message.
  • --> commonly indicates a return.
  • Add text to describe the reply content.
  • Requests and replies often come in pairs but are not required.
  • Declare participants first if you want clarity.
  • Time flows from top to bottom.

Sequence / BASICS / Self-Message

This is “talking to yourself,” which shows an internal step.

Why use it

  • Shows an object doing internal work without others.

How to use

  1. Alice -> Alice: Process

Possible use cases

  • Algorithm steps, internal calculations.

Practice (easy for kids)

  1. Practice 1: Change Alice to your name (e.g., Me) and change the message to something you want to do today.
  2. Practice 2: Change the action to “read data,” then add Me -> Me: check again.
  3. Practice 3: Change the message to “think,” then add Me -> Me: write key points.
  4. Practice 4: Change the name to Robot, then add Robot -> Robot: charge.
  5. Practice 5: Add Me -> Me: start at the top and Me -> Me: done at the end.

PlantUML syntax rules (complete list)

  • Both ends of the arrow can be the same participant.
  • Basic format: Role -> Role: Text.
  • Names can be used without prior declaration.
  • Text after the colon is optional.
  • Multiple self-messages show multiple internal steps.
  • Can pair with activate / deactivate.
  • Time flows from top to bottom.

Sequence / PARTICIPANTS / Colors & Styles

This example shows colors and styles to make the diagram clearer.

Why use it

  • Highlight important roles with color.

How to use

  1. Use skinparam to adjust styles.
  2. Add a color after a role: participant Alice #AliceBlue.

Possible use cases

  • Show the “core system” in a bright color.

Practice (easy for kids)

  1. Practice 1: Rename Alice and change the color to another one.
  2. Practice 2: Change the message to “start,” and make the other role darker.
  3. Practice 3: Add a third role and assign a different color.
  4. Practice 4: Change skinparam sequenceMessageAlign and adjust a message line.
  5. Practice 5: Use a style name (like #technology) for one role and change the message.

PlantUML syntax rules (complete list)

  • skinparam controls alignment, font, colors, etc.
  • Roles can be followed by a color or style name.
  • Colors can be names or hex, like #AliceBlue.
  • If the same role is declared again, the last one wins.
  • Style settings do not change message syntax.
