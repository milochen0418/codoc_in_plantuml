Sequence / BASICS / Simple Message

این مثال نشان می‌دهد «شخص A یک پیام به شخص B می‌فرستد»، مثل دادن یک یادداشت.

چرا استفاده کنیم

  • ترتیب گفتگو را ساده و قابل فهم نشان می‌دهد.
  • برای داستان‌های «چه کسی اول حرف می‌زند، بعد چه کسی جواب می‌دهد» عالی است.

چطور استفاده کنیم

  1. @startuml و @enduml را بنویسید.
  2. با Alice -> Bob: Message نشان دهید A با B صحبت می‌کند.

موارد استفادهٔ ممکن

  • جریان ورود: User → System → Server.
  • چت‌بات: شما می‌پرسید → ربات پاسخ می‌دهد.

تمرین (ساده برای کودکان)

  1. تمرین ۱: Alice و Bob را به نام‌های دلخواه تغییر دهید و پیام را به یک مکالمهٔ روزمره تبدیل کنید.
  2. تمرین ۲: ابتدا یک خط participant اضافه کنید، سپس پیام را به «اول سؤال، بعد جواب» تغییر دهید.
  3. تمرین ۳: پیام را به دو خط تقسیم کنید: اول سؤال، بعد جواب، و یکی از خطوط را کوتاه کنید.
  4. تمرین ۴: جای کسی که اول حرف می‌زند را عوض کنید و متن پیام را تغییر دهید.
  5. تمرین ۵: یک نفر سوم اضافه کنید که یک خط بگوید، سپس نام یکی را عوض کنید.

قواعد PlantUML (فهرست کامل)

  • با @startuml شروع و با @enduml تمام کنید.
  • شرکت‌کننده‌ها را می‌توان با participant تعریف کرد یا مستقیم استفاده کرد.
  • A -> B فراخوانی است، A --> B پاسخ است.
  • متن را بعد از کولن بگذارید: A -> B: Hello.
  • زمان از بالا به پایین می‌رود.
  • چند خط پیام یعنی گفت‌وگوی طولانی‌تر.
  • متن پیام اختیاری است: A -> B هم درست است.
  • نام‌های دارای فاصله را داخل کوتیشن بنویسید.

Sequence / BASICS / Return Message

این مثال «درخواست و پاسخ» را نشان می‌دهد، مثل سؤال و جواب.

چرا استفاده کنیم

  • مرحلهٔ درخواست و مرحلهٔ پاسخ را واضح نشان می‌دهد.

چطور استفاده کنیم

  1. A -> B درخواست است.
  2. B --> A پاسخ است.

موارد استفادهٔ ممکن

  • فراخوانی API: Frontend → Backend → بازگشت داده.

تمرین (ساده برای کودکان)

  1. تمرین ۱: نقش‌ها را به Client و Server تغییر دهید، پیام‌ها را به «دریافت داده / بازگرداندن داده» تبدیل کنید.
  2. تمرین ۲: درخواست را «query» و پاسخ را «query result» کنید.
  3. تمرین ۳: یک جفت درخواست/پاسخ دیگر اضافه کنید (در مجموع چهار خط) و متن خط دوم را تغییر دهید.
  4. تمرین ۴: شخص دیگری پاسخ بدهد و متن را تنظیم کنید.
  5. تمرین ۵: قبل از درخواست، یک self-message «آماده‌سازی» اضافه کنید، سپس درخواست را دقیق‌تر کنید.

قواعد PlantUML (فهرست کامل)

  • -> یک پیام می‌فرستد.
  • --> معمولاً پاسخ را نشان می‌دهد.
  • برای توضیح پاسخ، متن اضافه کنید.
  • درخواست و پاسخ اغلب جفت هستند ولی اجباری نیستند.
  • برای وضوح می‌توانید شرکت‌کننده‌ها را اول تعریف کنید.
  • زمان از بالا به پایین جریان دارد.

Sequence / BASICS / Self-Message

این یعنی «صحبت با خود»، که یک گام داخلی را نشان می‌دهد.

چرا استفاده کنیم

  • کار داخلی یک شیء را بدون دیگران نشان می‌دهد.

چطور استفاده کنیم

  1. Alice -> Alice: Process

موارد استفادهٔ ممکن

  • گام‌های الگوریتم، محاسبات داخلی.

تمرین (ساده برای کودکان)

  1. تمرین ۱: Alice را به نام خودتان (مثلاً Me) تغییر دهید و پیام را به کاری که امروز می‌خواهید انجام دهید تبدیل کنید.
  2. تمرین ۲: عمل را «خواندن داده» کنید، سپس Me -> Me: دوباره بررسی اضافه کنید.
  3. تمرین ۳: پیام را «فکر کردن» کنید، سپس Me -> Me: نکات مهم را بنویس اضافه کنید.
  4. تمرین ۴: نام را Robot کنید، سپس Robot -> Robot: شارژ اضافه کنید.
  5. تمرین ۵: در ابتدا Me -> Me: start و در انتها Me -> Me: done اضافه کنید.

قواعد PlantUML (فهرست کامل)

  • هر دو سر پیکان می‌تواند یک شرکت‌کننده باشد.
  • فرمت پایه: Role -> Role: Text.
  • نام‌ها می‌توانند بدون تعریف قبلی استفاده شوند.
  • متن بعد از کولن اختیاری است.
  • چند self-message یعنی چند گام داخلی.
  • می‌تواند با activate / deactivate همراه شود.
  • زمان از بالا به پایین می‌رود.

Sequence / PARTICIPANTS / Types

این مثال آیکون‌های متفاوت برای نقش‌های مختلف را نشان می‌دهد، مثل شخص، سیستم، دیتابیس.

چرا استفاده کنیم

  • آیکون‌ها نقش‌ها را سریع‌تر قابل تشخیص می‌کنند.

چطور استفاده کنیم

  1. از actor، boundary، control، database، entity استفاده کنید.

موارد استفادهٔ ممکن

  • جریان ورود: User → Web → Logic → Database.

تمرین (ساده برای کودکان)

  1. تمرین ۱: User را به Student تغییر دهید و پیام را به «ورود تکلیف» تبدیل کنید.
  2. تمرین ۲: boundary App اضافه کنید، سپس پیکان را اول به App بفرستید.
  3. تمرین ۳: database DB را به database Books تغییر دهید و پیام را عوض کنید.
  4. تمرین ۴: یک نقش control اضافه کنید و یک خط تعامل بنویسید.
  5. تمرین ۵: ترتیب شرکت‌کننده‌ها را عوض کنید و پیام‌ها را تنظیم کنید.

قواعد PlantUML (فهرست کامل)

  • نوع نقش ظاهر را عوض می‌کند، نه نام را.
  • برای تعامل از -> استفاده کنید.
  • فرمت تعریف: actor/boundary/control/database/entity Name.
  • نقش‌ها را اول تعریف کنید، سپس نامشان را به کار ببرید.
  • نام‌های تعریف‌نشده شرکت‌کنندهٔ عادی هستند.

Sequence / PARTICIPANTS / Colors & Styles

این مثال رنگ‌ها و سبک‌ها را برای واضح‌تر شدن نمودار نشان می‌دهد.

چرا استفاده کنیم

  • نقش‌های مهم با رنگ برجسته می‌شوند.

چطور استفاده کنیم

  1. برای تنظیم سبک از skinparam استفاده کنید.
  2. بعد از نقش رنگ اضافه کنید: participant Alice #AliceBlue.

موارد استفادهٔ ممکن

  • نشان دادن «سیستم مرکزی» با رنگ روشن.

تمرین (ساده برای کودکان)

  1. تمرین ۱: نام Alice را عوض کنید و رنگ دیگری بگذارید.
  2. تمرین ۲: پیام را «شروع» کنید و نقش دیگر را تیره‌تر کنید.
  3. تمرین ۳: نقش سوم اضافه کنید و رنگ متفاوت بدهید.
  4. تمرین ۴: skinparam sequenceMessageAlign را تغییر دهید و یک خط پیام را تنظیم کنید.
  5. تمرین ۵: از نام سبک (مثل #technology) برای یک نقش استفاده کنید و پیام را تغییر دهید.

قواعد PlantUML (فهرست کامل)

  • skinparam تراز، فونت، رنگ و ... را کنترل می‌کند.
  • نقش‌ها می‌توانند رنگ یا نام سبک داشته باشند.
  • رنگ‌ها می‌توانند نام یا hex باشند، مثل #AliceBlue.
  • اگر یک نقش دوباره تعریف شود، آخرین مورد اعمال می‌شود.
  • تنظیمات سبک نحو پیام را تغییر نمی‌دهد.

Sequence / CONTROL FLOW / Alt/Else (Choices)

این مثال «اگر... وگرنه...» را مثل چند انتخاب نشان می‌دهد.

چرا استفاده کنیم

  • شاخه‌های مختلف برای شرط‌های مختلف را نشان می‌دهد.

چطور استفاده کنیم

  1. با alt شروع کنید، برای شاخهٔ دیگر از else استفاده کنید، و با end تمام کنید.

موارد استفادهٔ ممکن

  • پرداخت موفق یا ناموفق.

تمرین (ساده برای کودکان)

  1. تمرین ۱: شرط‌ها را به «پول داریم / پول نداریم» تغییر دهید و پیام‌ها را عوض کنید.
  2. تمرین ۲: یک شاخهٔ else سوم اضافه کنید و یک پیام بنویسید.
  3. تمرین ۳: عنوان alt را به سناریوی خودتان تغییر دهید و پیام‌ها را ویرایش کنید.
  4. تمرین ۴: در هر شاخه یک خط اضافه کنید و عنوان‌ها را تغییر دهید.
  5. تمرین ۵: یک شاخه «تلاش دوباره بعد از خطا» بسازید و شاخهٔ دیگر را به پایان متفاوت تبدیل کنید.

قواعد PlantUML (فهرست کامل)

  • alt Condition یک شاخه را شروع می‌کند.
  • می‌تواند چندین شاخهٔ else داشته باشد.
  • با end تمام کنید.
  • هر شاخه می‌تواند چند پیام داشته باشد.
  • متن شرط اختیاری است ولی بهتر است باشد.

Sequence / CONTROL FLOW / Loop

این مثال کارهای تکراری را نشان می‌دهد.

چرا استفاده کنیم

  • گام‌های تکراری را نشان می‌دهد.

چطور استفاده کنیم

  1. خطوط تکراری را داخل loop و end قرار دهید.

موارد استفادهٔ ممکن

  • هر ثانیه وضعیت را چک کردن.

تمرین (ساده برای کودکان)

  1. تمرین ۱: برچسب لوپ را «هر ۳ بار» بگذارید و پیام را تغییر دهید.
  2. تمرین ۲: یک پیام دیگر داخل لوپ اضافه کنید و پیام اصلی را کوتاه‌تر کنید.
  3. تمرین ۳: نام نقش‌ها را عوض کنید و پیام را «بررسی» بگذارید.
  4. تمرین ۴: برچسب لوپ را به شرطی مثل «تا وقتی تمام شود» تغییر دهید و پیام‌ها را تنظیم کنید.
  5. تمرین ۵: پیام‌های شروع/پایان را بیرون از لوپ اضافه کنید و متن‌ها را سفارشی کنید.

قواعد PlantUML (فهرست کامل)

  • loop count/condition می‌تواند در همان خط بیاید.
  • end بلوک لوپ را تمام می‌کند.
  • چندین پیام می‌تواند داخل لوپ باشد.

Sequence / CONTROL FLOW / Parallel (Par)

این مثال دو کار هم‌زمان را نشان می‌دهد.

چرا استفاده کنیم

  • کارهای موازی را نشان می‌دهد.

چطور استفاده کنیم

  1. با par شروع کنید، برای شاخهٔ موازی دیگر از else استفاده کنید، و با end تمام کنید.

موارد استفادهٔ ممکن

  • اطلاع دادن هم‌زمان به دو سرویس.

تمرین (ساده برای کودکان)

  1. تمرین ۱: شاخه‌ها را به «ارسال ایمیل» و «ارسال SMS» تغییر دهید و نقش‌ها را عوض کنید.
  2. تمرین ۲: نقش‌ها را عوض کنید و در هر شاخه یک خط دیگر اضافه کنید.
  3. تمرین ۳: یک شاخهٔ سوم (else) اضافه کنید و یک پیام بنویسید، سپس یکی از خطوط را کوتاه کنید.
  4. تمرین ۴: یکی از شاخه‌ها را «آهسته انجام بده» کنید، یک پیام پایانی اضافه کنید و متن شاخهٔ دیگر را تنظیم کنید.
  5. تمرین ۵: قبل از par پیام «شروع کار» و بعد از end پیام «تمام شد» اضافه کنید.

قواعد PlantUML (فهرست کامل)

  • par یک بلوک موازی می‌سازد.
  • هر شاخه می‌تواند دنبالهٔ پیام خودش را داشته باشد.
  • else به شاخهٔ موازی دیگر می‌رود.
  • با end

Sequence / ADVANCED / Activation

این مثال با نوارهای activation نشان می‌دهد چه زمانی یک نقش مشغول است.

چرا استفاده کنیم

  • نشان می‌دهد چه کسی مشغول کار است و چقدر طول می‌کشد.

چطور استفاده کنیم

  1. برای شروع از activate Role استفاده کنید.
  2. برای پایان از deactivate Role استفاده کنید.

موارد استفادهٔ ممکن

  • نشان دادن زمان پردازش سرور.

تمرین (ساده برای کودکان)

  1. تمرین ۱: نقش را Server کنید و پیام را «processing» بگذارید.
  2. تمرین ۲: یک خط درخواست اضافه کنید، سپس activate را بلافاصله بعد از درخواست بگذارید.
  3. تمرین ۳: یک پیام میانی مانند «نیمه‌راه» اضافه کنید و با متن خودتان جایگزین کنید.
  4. تمرین ۴: نقش دیگری با activation و پیام خودش اضافه کنید.
  5. تمرین ۵: شروع/پایان activation را یک خط پایین‌تر ببرید و پیام را تنظیم کنید.

قواعد PlantUML (فهرست کامل)

  • می‌توانید activate/deactivate را قبل یا بعد از پیام‌ها قرار دهید.
  • activate و deactivate باید جفت باشند.
  • وقتی نقش فعال است، می‌توان پیام اضافه کرد.

Sequence / ADVANCED / Notes

این مثال نحوهٔ افزودن یادداشت‌های چسبان را نشان می‌دهد.

چرا استفاده کنیم

  • برای شفافیت توضیح اضافه می‌کند.

چطور استفاده کنیم

  1. از note left یا note right استفاده کنید.
  2. یادداشت را با end note تمام کنید.

موارد استفادهٔ ممکن

  • یادآوری مثل «این مرحله ممکن است خطا بدهد».

تمرین (ساده برای کودکان)

  1. تمرین ۱: متن یادداشت را به یادداشت خودتان تغییر دهید.
  2. تمرین ۲: یادداشت را به سمت راست ببرید و متن را تغییر دهید.
  3. تمرین ۳: کنار نقش دیگری یک یادداشت دوم اضافه کنید.
  4. تمرین ۴: یادداشت را دوخطی کنید و در خط دوم دلیل بنویسید.
  5. تمرین ۵: یادداشت را بعد از یک پیام دیگر منتقل کنید.

قواعد PlantUML (فهرست کامل)

  • یادداشت‌ها می‌توانند left، right یا over باشند.
  • یادداشت‌های چندخطی از note و end note استفاده می‌کنند.
  • یادداشت‌ها جریان پیام را تغییر نمی‌دهند.

Use Case / BASICS / Simple Use Case

نمودار Use Case فهرستی از «کارهایی که سیستم می‌تواند انجام دهد» است.

چرا استفاده کنیم

  • عملکردهایی که کاربر نیاز دارد را واضح فهرست می‌کند.

چطور استفاده کنیم

  1. برای کاربر از actor استفاده کنید.
  2. برای یک عملکرد از (UseCase) استفاده کنید.

موارد استفادهٔ ممکن

  • ورود، جست‌وجو، پرداخت و ...

تمرین (ساده برای کودکان)

  1. تمرین ۱: کاربر را Teacher و Use Case را «create quiz» کنید.
  2. تمرین ۲: یک Use Case دیگر اضافه کنید.
  3. تمرین ۳: نام‌های Use Case و actor را عوض کنید.
  4. تمرین ۴: actor دوم اضافه کنید و به Use Case وصل کنید.
