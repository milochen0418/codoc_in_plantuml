Sequence / BASICS / Simple Message

Este ejemplo muestra “la persona A envía un mensaje a la persona B”, como pasar una nota.

Por qué usarlo

  • Muestra el orden de una conversación de forma clara.
  • Ideal para historias de “quién habla primero y quién responde después”.

Cómo usarlo

  1. Escribe @startuml y @enduml.
  2. Usa Alice -> Bob: Mensaje para mostrar que A habla con B.

Posibles usos

  • Inicio de sesión: Usuario → Sistema → Servidor.
  • Chatbot: Tú preguntas → Bot responde.

Práctica (fácil para niños)

  1. Práctica 1: Cambia Alice y Bob por tus nombres favoritos y cambia el mensaje por una conversación diaria.
  2. Práctica 2: Agrega una línea participant primero y cambia el mensaje a “preguntar primero, responder después”.
  3. Práctica 3: Divide el mensaje en dos líneas: pregunta y respuesta, y acorta una línea.
  4. Práctica 4: Cambia quién habla primero y ajusta el texto del mensaje.
  5. Práctica 5: Agrega un tercer personaje con un mensaje y renombra a uno.

Reglas de sintaxis PlantUML (lista completa)

  • Empieza con @startuml y termina con @enduml.
  • Los participantes pueden declararse con participant o usarse directamente.
  • A -> B es una llamada, A --> B es un retorno.
  • El texto va después de dos puntos: A -> B: Hola.
  • El tiempo va de arriba hacia abajo.
  • Múltiples mensajes muestran una conversación más larga.
  • El texto del mensaje puede omitirse: A -> B es válido.
  • Los nombres con espacios van entre comillas.

Sequence / BASICS / Return Message

Este ejemplo muestra “solicitud y respuesta”, como una pregunta y una respuesta.

Por qué usarlo

  • Deja claros los pasos de solicitud y respuesta.

Cómo usarlo

  1. A -> B es la solicitud.
  2. B --> A es la respuesta.

Posibles usos

  • Llamada API: Frontend → Backend → Retorno de datos.

Práctica (fácil para niños)

  1. Práctica 1: Cambia roles a Client y Server y mensajes a “obtener datos / devolver datos”.
  2. Práctica 2: Cambia la solicitud a “consulta” y la respuesta a “resultado”.
  3. Práctica 3: Agrega otra solicitud y respuesta (cuatro líneas) para mostrar dos preguntas y cambia el segundo texto.
  4. Práctica 4: Haz que otra persona responda y ajusta el texto.
  5. Práctica 5: Agrega un mensaje propio “preparando” antes de la solicitud y hazla más específica.

Reglas de sintaxis PlantUML (lista completa)

  • -> envía un mensaje.
  • --> suele indicar un retorno.
  • Agrega texto para describir el contenido de la respuesta.
  • Las solicitudes y respuestas suelen venir en pares, pero no es obligatorio.
  • Declara participantes primero si quieres claridad.
  • El tiempo fluye de arriba hacia abajo.

Sequence / BASICS / Self-Message

Esto es “hablar contigo mismo”, que muestra un paso interno.

Por qué usarlo

  • Muestra que un objeto hace trabajo interno sin otros.

Cómo usarlo

  1. Alice -> Alice: Process

Posibles usos

  • Pasos de algoritmo, cálculos internos.

Práctica (fácil para niños)

  1. Práctica 1: Cambia Alice por tu nombre (por ejemplo, Me) y cambia el mensaje por algo que quieras hacer hoy.
  2. Práctica 2: Cambia la acción a “leer datos”, y agrega Me -> Me: revisar otra vez.
  3. Práctica 3: Cambia el mensaje a “pensar”, y agrega Me -> Me: anotar puntos clave.
  4. Práctica 4: Cambia el nombre a Robot y agrega Robot -> Robot: cargar.
  5. Práctica 5: Agrega Me -> Me: empezar al inicio y Me -> Me: terminado al final.

Reglas de sintaxis PlantUML (lista completa)

  • Ambos extremos de la flecha pueden ser el mismo participante.
  • Formato básico: Rol -> Rol: Texto.
  • Los nombres pueden usarse sin declaración previa.
  • El texto después de los dos puntos es opcional.
  • Varios mensajes propios muestran múltiples pasos internos.
  • Se puede combinar con activate / deactivate.
  • El tiempo fluye de arriba hacia abajo.

Sequence / PARTICIPANTS / Types

Este ejemplo muestra distintos íconos para distintos roles: persona, sistema, base de datos.

Por qué usarlo

  • Los íconos ayudan a reconocer roles más rápido.

Cómo usarlo

  1. Usa actor, boundary, control, database, entity.

Posibles usos

  • Inicio de sesión: Usuario → Web → Lógica → Base de datos.

Práctica (fácil para niños)

  1. Práctica 1: Cambia User a Student y el mensaje a “iniciar tarea”.
  2. Práctica 2: Agrega boundary App y dirige la flecha primero a App.
  3. Práctica 3: Cambia database DB a database Books y ajusta el mensaje.
  4. Práctica 4: Agrega un rol control y una línea de interacción.
  5. Práctica 5: Reordena participantes y ajusta mensajes.

Reglas de sintaxis PlantUML (lista completa)

  • El tipo de rol cambia la apariencia, no el nombre.
  • Usa -> para interacciones.
  • Formato de declaración: actor/boundary/control/database/entity Nombre.
  • Declara roles primero y luego usa sus nombres.
  • Los nombres no declarados se tratan como participantes normales.

Sequence / PARTICIPANTS / Colors & Styles

Este ejemplo muestra colores y estilos para hacer el diagrama más claro.

Por qué usarlo

  • Resalta roles importantes con color.

Cómo usarlo

  1. Usa skinparam para ajustar estilos.
  2. Agrega un color después del rol: participant Alice #AliceBlue.

Posibles usos

  • Mostrar el “sistema central” en un color brillante.

Práctica (fácil para niños)

  1. Práctica 1: Cambia Alice y el color por otro.
  2. Práctica 2: Cambia el mensaje a “inicio” y oscurece el otro rol.
  3. Práctica 3: Agrega un tercer rol y asigna un color distinto.
  4. Práctica 4: Cambia skinparam sequenceMessageAlign y ajusta un mensaje.
  5. Práctica 5: Usa un nombre de estilo (como #technology) y cambia el mensaje.

Reglas de sintaxis PlantUML (lista completa)

  • skinparam controla alineación, fuente, colores, etc.
  • Los roles pueden ir seguidos de un color o un estilo.
  • Los colores pueden ser nombres o hex, como #AliceBlue.
  • Si el mismo rol se declara otra vez, prevalece el último.
  • Los estilos no cambian la sintaxis de mensajes.

Sequence / CONTROL FLOW / Alt/Else (Choices)

Este ejemplo muestra “si…sino…”, como múltiples opciones.

Por qué usarlo

  • Muestra distintas ramas según condiciones.

Cómo usarlo

  1. Usa alt para empezar, else para otra rama y end para terminar.

Posibles usos

  • Pago exitoso o fallido.

Práctica (fácil para niños)

  1. Práctica 1: Cambia condiciones a “con dinero / sin dinero” y ajusta mensajes.
  2. Práctica 2: Agrega una tercera rama else y un mensaje.
  3. Práctica 3: Cambia el título de alt a tu escenario y edita mensajes.
  4. Práctica 4: Agrega una línea en cada rama y cambia los títulos.
  5. Práctica 5: Haz una rama “intentar de nuevo” y otra con un final distinto.

Reglas de sintaxis PlantUML (lista completa)

  • alt Condición inicia una rama.
  • Pueden existir varias ramas else.
  • Termina con end.
  • Cada rama puede tener varios mensajes.
  • El texto de la condición es opcional, pero recomendable.
