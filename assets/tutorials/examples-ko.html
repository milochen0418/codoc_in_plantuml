Sequence / BASICS / Simple Message

이 예제는 “A가 B에게 메시지를 보낸다”를 보여 줍니다. 메모 전달과 비슷합니다.

왜 사용하나요

  • 대화의 순서를 간단히 보여 줍니다.
  • “누가 먼저 말하고, 누가 나중에 답하는지” 같은 이야기 설명에 좋습니다.

어떻게 사용하나요

  1. @startuml과 @enduml을 적습니다.
  2. Alice -> Bob: Message로 A가 B에게 말하는 것을 표현합니다.

가능한 사용 사례

  • 로그인 흐름: 사용자 → 시스템 → 서버.
  • 챗봇: 내가 질문 → 봇이 답변.

연습(아이들도 쉬운)

  1. 연습 1: Alice와 Bob을 좋아하는 이름으로 바꾸고, 문장을 일상 대화로 바꿔 보세요.
  2. 연습 2: 먼저 participant 한 줄을 추가하고 “먼저 질문, 나중에 답변”으로 바꿔 보세요.
  3. 연습 3: 메시지를 두 줄로 나누세요: 질문 → 답변, 그리고 한 줄을 줄여 보세요.
  4. 연습 4: 먼저 말하는 사람을 바꾸고 텍스트를 수정하세요.
  5. 연습 5: 세 번째 사람을 추가하고 누군가의 이름을 바꿔 보세요.

PlantUML 문법 규칙(전체 목록)

  • @startuml로 시작하고 @enduml로 끝냅니다.
  • 참여자는 participant로 선언하거나 바로 사용할 수 있습니다.
  • A -> B는 호출, A --> B는 응답입니다.
  • 텍스트는 콜론 뒤에 씁니다: A -> B: Hello.
  • 시간은 위에서 아래로 흐릅니다.
  • 메시지가 많을수록 대화가 길어집니다.
  • 메시지 텍스트는 선택 사항입니다: A -> B도 유효합니다.
  • 공백이 있는 이름은 따옴표로 감쌉니다.

Sequence / BASICS / Return Message

이 예제는 “요청과 응답”을 보여 줍니다. 질문과 답변처럼요.

왜 사용하나요

  • 요청과 응답의 단계를 분명히 보여 줍니다.

어떻게 사용하나요

  1. A -> B는 요청입니다.
  2. B --> A는 응답입니다.

가능한 사용 사례

  • API 호출: 프론트엔드 → 백엔드 → 데이터 반환.

연습(아이들도 쉬운)

  1. 연습 1: 역할을 Client와 Server로 바꾸고 메시지를 “데이터 요청 / 데이터 반환”으로 바꿔 보세요.
  2. 연습 2: 요청을 “query”, 응답을 “query result”로 바꿔 보세요.
  3. 연습 3: 요청‑응답을 한 쌍 더 추가(4줄)하고 두 번째 텍스트를 바꿔 보세요.
  4. 연습 4: 다른 사람이 응답하도록 하고 텍스트를 조정하세요.
  5. 연습 5: 요청 전에 “준비” self‑message를 추가하세요.

PlantUML 문법 규칙(전체 목록)

  • ->는 메시지 전송입니다.
  • -->는 보통 응답을 뜻합니다.
  • 응답의 내용을 설명하는 텍스트를 추가하세요.
  • 요청과 응답은 종종 쌍이지만 꼭 그래야 하는 것은 아닙니다.
  • 명확성을 위해 먼저 참여자를 선언해도 됩니다.
  • 시간은 위에서 아래로 흐릅니다.

Sequence / BASICS / Self-Message

“혼잣말”처럼 내부 단계를 보여 줍니다.

왜 사용하나요

  • 객체가 내부적으로 작업하는 모습을 보여 줍니다.

어떻게 사용하나요

  1. Alice -> Alice: Process

가능한 사용 사례

  • 알고리즘 단계, 내부 계산.

연습(아이들도 쉬운)

  1. 연습 1: Alice를 내 이름(예: Me)으로 바꾸고 메시지를 바꾸세요.
  2. 연습 2: 행동을 “데이터 읽기”로 바꾸고 Me -> Me: 다시 확인을 추가하세요.
  3. 연습 3: 메시지를 “생각하기”로 바꾸고 Me -> Me: 핵심 메모를 추가하세요.
  4. 연습 4: 이름을 Robot으로 바꾸고 Robot -> Robot: 충전을 추가하세요.
  5. 연습 5: 처음에 Me -> Me: start, 끝에 Me -> Me: done을 추가하세요.

PlantUML 문법 규칙(전체 목록)

  • 화살표의 양끝이 같은 참여자일 수 있습니다.
  • 기본 형식: Role -> Role: Text.
  • 이름은 선언 없이도 사용할 수 있습니다.
  • 콜론 뒤의 텍스트는 선택 사항입니다.
  • 여러 self‑message는 더 많은 내부 단계를 보여 줍니다.
  • activate / deactivate와 함께 사용할 수 있습니다.
  • 시간은 위에서 아래로 흐릅니다.

Sequence / PARTICIPANTS / Types

이 예제는 사람, 시스템, 데이터베이스 등 역할에 따라 아이콘이 달라짐을 보여 줍니다.

왜 사용하나요

  • 아이콘으로 역할을 더 빠르게 구분할 수 있습니다.

어떻게 사용하나요

  1. actor, boundary, control, database, entity를 사용합니다.

가능한 사용 사례

  • 로그인 흐름: User → Web → Logic → Database.

연습(아이들도 쉬운)

  1. 연습 1: User를 Student로 바꾸고 메시지를 수정하세요.
  2. 연습 2: boundary App을 추가하고 화살표를 먼저 App으로 보내세요.
  3. 연습 3: database DB를 database Books로 바꾸고 메시지를 수정하세요.
  4. 연습 4: control 역할을 추가하고 한 줄의 상호작용을 넣으세요.
  5. 연습 5: 참여자 순서를 바꾸고 메시지를 조정하세요.

PlantUML 문법 규칙(전체 목록)

  • 역할 타입은 모양만 바꾸고 이름은 바꾸지 않습니다.
  • 상호작용에는 ->를 사용합니다.
  • 선언 형식: actor/boundary/control/database/entity Name.
  • 먼저 역할을 선언한 뒤 이름을 사용하세요.
  • 선언되지 않은 이름은 일반 참여자입니다.

Sequence / PARTICIPANTS / Colors & Styles

이 예제는 색상과 스타일로 다이어그램을 더 명확하게 만드는 방법을 보여 줍니다.

왜 사용하나요

  • 중요한 역할을 색으로 강조합니다.

어떻게 사용하나요

  1. 스타일을 조정하려면 skinparam을 사용합니다.
  2. 역할 뒤에 색을 붙입니다: participant Alice #AliceBlue.

가능한 사용 사례

  • “중앙 시스템”을 밝은 색으로 강조.

연습(아이들도 쉬운)

  1. 연습 1: Alice 이름을 바꾸고 색을 변경하세요.
  2. 연습 2: 메시지를 “start”로 바꾸고 다른 역할을 더 어둡게 하세요.
  3. 연습 3: 세 번째 역할을 추가하고 다른 색을 주세요.
  4. 연습 4: skinparam sequenceMessageAlign을 바꾸고 한 줄을 수정하세요.
  5. 연습 5: 스타일 이름(예: #technology)을 쓰고 메시지를 바꾸세요.

PlantUML 문법 규칙(전체 목록)

  • skinparam은 정렬, 글꼴, 색상 등을 제어합니다.
  • 역할에 색 또는 스타일 이름을 줄 수 있습니다.
  • 색상은 이름 또는 hex가 될 수 있습니다(예: #AliceBlue).
  • 같은 역할을 다시 선언하면 마지막 선언이 적용됩니다.
  • 스타일 설정은 메시지 문법을 바꾸지 않습니다.

Sequence / CONTROL FLOW / Alt/Else (Choices)

“만약… 아니면…” 같은 선택을 보여 줍니다.

왜 사용하나요

  • 조건에 따른 서로 다른 가지를 보여 줍니다.

어떻게 사용하나요

  1. alt로 시작, else로 다른 가지, end로 종료합니다.

가능한 사용 사례

  • 결제 성공/실패.

연습(아이들도 쉬운)

  1. 연습 1: 조건을 “돈 있음 / 돈 없음”으로 바꾸고 메시지를 수정하세요.
  2. 연습 2: 세 번째 else 가지와 메시지 하나를 추가하세요.
  3. 연습 3: alt 제목을 내 시나리오로 바꾸세요.
  4. 연습 4: 각 가지에 한 줄씩 더하고 제목을 바꾸세요.
  5. 연습 5: “오류 후 재시도” 가지를 추가하세요.

PlantUML 문법 규칙(전체 목록)

  • alt Condition이 가지를 시작합니다.
  • 여러 else 가지가 있을 수 있습니다.
  • end로 닫습니다.
  • 각 가지에는 여러 메시지가 들어갈 수 있습니다.
  • 조건 텍스트는 선택 사항이지만 권장됩니다.

왜 사용하나요

  • 반복되는 단계를 보여 줍니다.

어떻게 사용하나요

가능한 사용 사례

  • 매초 상태를 확인.

연습(아이들도 쉬운)

  1. 연습 1: 루프 라벨을 “매 3번”으로 바꾸고 메시지를 수정하세요.

Sequence / CONTROL FLOW / Parallel (Par)

동시에 일어나는 두 일을 보여 줍니다.

왜 사용하나요

  • 병렬 작업을 보여 줍니다.

어떻게 사용하나요

  1. par로 시작, 다른 병렬 가지는 else, 종료는 end.

가능한 사용 사례

  • 두 서비스를 동시에 알림.

연습(아이들도 쉬운)

  1. 연습 1: 가지를 “이메일 보내기”와 “SMS 보내기”로 바꾸고 역할 이름을 수정하세요.
  2. 연습 2: 역할을 바꾸고 각 가지에 한 줄씩 추가하세요.
  3. 연습 3: 세 번째 가지(else)와 메시지를 추가하세요.
  4. 연습 4: 한 가지를 “천천히 진행”으로 바꾸고 마지막 메시지를 추가하세요.
  5. 연습 5: par 앞에 “작업 시작”, end 뒤에 “완료”를 추가하세요.

PlantUML 문법 규칙(전체 목록)

  • par는 병렬 블록을 만듭니다.
  • 각 가지는 자신만의 메시지 흐름을 가질 수 있습니다.
  • else는 다른 병렬 가지로 이동합니다.
  • end로 닫습니다.

Use Case / BASICS / Simple Use Case

유스케이스 다이어그램은 “시스템이 할 수 있는 일”의 목록입니다.

왜 사용하나요

  • 사용자 기능을 명확히 나열합니다.

어떻게 사용하나요

  1. 사용자는 actor로 나타냅니다.
  2. 기능은 (UseCase)로 나타냅니다.

가능한 사용 사례

  • 로그인, 검색, 결제 등.

연습(아이들도 쉬운)

Use Case / RELATIONSHIPS / Include/Extend

유스케이스 간 include/extend 관계를 보여 줍니다.

왜 사용하나요

  • 공통 단계나 선택 단계를 표시합니다.

어떻게 사용하나요
