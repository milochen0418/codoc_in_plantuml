Sequence / BASICS / Simple Message

هذا المثال يوضح “الشخص A يرسل رسالة واحدة إلى الشخص B”، مثل تمرير ورقة.

لماذا نستخدمه

  • يعرض ترتيب الحوار بوضوح.
  • ممتاز لقصص “من يتحدث أولًا ومن يرد بعد ذلك”.

كيفية الاستخدام

  1. اكتب @startuml و @enduml.
  2. استخدم Alice -> Bob: Message لإظهار أن A يتحدث إلى B.

استخدامات ممكنة

  • تدفق تسجيل الدخول: المستخدم → النظام → الخادم.
  • روبوت الدردشة: أنت تسأل → الروبوت يجيب.

تمارين (سهلة للأطفال)

  1. تمرين 1: غيّر Alice و Bob إلى أسماء تحبها، ثم غيّر الرسالة إلى حوار يومي.
  2. تمرين 2: أضف سطر participant أولًا، ثم غيّر الرسالة إلى “اسأل أولًا ثم أجب”.
  3. تمرين 3: حوّل الرسالة إلى سطرين: سؤال ثم جواب، ثم اختصر أحد السطرين.
  4. تمرين 4: بدّل من يتحدث أولًا، ثم عدّل نص الرسالة.
  5. تمرين 5: أضف شخصًا ثالثًا برسالة واحدة، ثم غيّر اسم أحد الأشخاص.

قواعد PlantUML (قائمة كاملة)

  • ابدأ بـ @startuml وأنهِ بـ @enduml.
  • يمكن تعريف المشاركين بـ participant أو استخدام الأسماء مباشرة.
  • A -> B رسالة استدعاء، و A --> B رسالة عودة.
  • ضع النص بعد النقطتين: A -> B: Hello.
  • الزمن من الأعلى إلى الأسفل.
  • عدة رسائل تمثل حوارًا أطول.
  • يمكن حذف نص الرسالة: A -> B صالح.
  • الأسماء التي تحتوي على فراغات تُكتب بين علامتي اقتباس.

Sequence / BASICS / Return Message

هذا المثال يوضح “طلب واستجابة”، مثل سؤال وجواب.

لماذا نستخدمه

  • يوضح خطوات الطلب والرد بوضوح.

كيفية الاستخدام

  1. A -> B هو الطلب.
  2. B --> A هو الاستجابة.

استخدامات ممكنة

  • استدعاء API: الواجهة → الخلفية → إرجاع البيانات.

تمارين (سهلة للأطفال)

  1. تمرين 1: غيّر الأدوار إلى Client و Server، والرسائل إلى “احصل على البيانات / أعد البيانات”.
  2. تمرين 2: غيّر الطلب إلى “استعلام”، والرد إلى “نتيجة الاستعلام”.
  3. تمرين 3: أضف طلبًا وردًا آخرين (أربعة أسطر) لعرض سؤالين، ثم عدّل نص السؤال الثاني.
  4. تمرين 4: اجعل شخصًا آخر يرد، وعدّل النص.
  5. تمرين 5: أضف رسالة ذاتية “جارٍ التحضير” قبل الطلب، ثم اجعل الطلب أكثر تحديدًا.

قواعد PlantUML (قائمة كاملة)

  • -> لإرسال الرسالة.
  • --> غالبًا للرسالة الراجعة.
  • يمكن إضافة نص لوصف محتوى الرد.
  • الطلبات والردود تأتي عادةً في أزواج لكنها ليست إلزامية.
  • يمكن تعريف المشاركين أولًا لزيادة الوضوح.
  • الزمن من الأعلى إلى الأسفل.

Sequence / BASICS / Self-Message

هذا يعني “حديث الذات”، أي خطوة داخلية.

لماذا نستخدمه

  • يبين أن الكائن ينفذ عملًا داخليًا دون الآخرين.

كيفية الاستخدام

  1. Alice -> Alice: Process

استخدامات ممكنة

  • خطوات خوارزمية، حسابات داخلية.

تمارين (سهلة للأطفال)

  1. تمرين 1: غيّر Alice إلى اسمك (مثل Me)، وعدّل الرسالة إلى شيء تريد فعله اليوم.
  2. تمرين 2: غيّر الفعل إلى “اقرأ البيانات”، ثم أضف Me -> Me: تحقق مرة أخرى.
  3. تمرين 3: غيّر الرسالة إلى “فكّر”، ثم أضف Me -> Me: اكتب النقاط المهمة.
  4. تمرين 4: غيّر الاسم إلى Robot ثم أضف Robot -> Robot: اشحن.
  5. تمرين 5: أضف Me -> Me: ابدأ في الأعلى و Me -> Me: انتهى في النهاية.

قواعد PlantUML (قائمة كاملة)

  • يمكن أن يكون طرفا السهم نفس المشارك.
  • الصيغة الأساسية: Role -> Role: Text.
  • يمكن استخدام الأسماء دون تعريف مسبق.
  • النص بعد النقطتين اختياري.
  • عدة رسائل ذاتية تعني عدة خطوات داخلية.
  • يمكن دمجه مع activate / deactivate.
  • الزمن من الأعلى إلى الأسفل.

Sequence / PARTICIPANTS / Types

هذا المثال يوضح أيقونات مختلفة للأدوار (شخص، نظام، قاعدة بيانات).

لماذا نستخدمه

  • الأيقونات تساعد على التعرف على الأدوار بسرعة.

كيفية الاستخدام

  1. استخدم actor و boundary و control و database و entity.

استخدامات ممكنة

  • تسجيل الدخول: المستخدم → الويب → المنطق → قاعدة البيانات.

تمارين (سهلة للأطفال)

  1. تمرين 1: غيّر User إلى Student ورسالة “تسجيل الدخول للواجب”.
  2. تمرين 2: أضف boundary App واجعل السهم يذهب إلى App أولًا.
  3. تمرين 3: غيّر database DB إلى database Books وعدّل الرسالة.
  4. تمرين 4: أضف دور control وسطر تفاعل واحد.
  5. تمرين 5: أعد ترتيب المشاركين وعدّل الرسائل تبعًا لذلك.

قواعد PlantUML (قائمة كاملة)

  • نوع الدور يغيّر الشكل فقط وليس الاسم.
  • يمكن استخدام -> للتفاعل.
  • صيغة التعريف: actor/boundary/control/database/entity Name.
  • عرّف الأدوار أولًا ثم استخدم الأسماء.
  • الأسماء غير المعرّفة تعتبر مشاركين عاديين.

Sequence / PARTICIPANTS / Colors & Styles

هذا المثال يوضح الألوان والأنماط لجعل الرسم أوضح.

لماذا نستخدمه

  • تمييز الأدوار المهمة بالألوان.

كيفية الاستخدام

  1. استخدم skinparam لتعديل الأنماط.
  2. أضف لونًا بعد الدور: participant Alice #AliceBlue.

استخدامات ممكنة

  • إظهار “النظام الأساسي” بلون بارز.

تمارين (سهلة للأطفال)

  1. تمرين 1: غيّر Alice واللون إلى لون آخر.
  2. تمرين 2: غيّر الرسالة إلى “ابدأ” واجعل الدور الآخر أغمق.
  3. تمرين 3: أضف دورًا ثالثًا بلون مختلف.
  4. تمرين 4: غيّر skinparam sequenceMessageAlign وعدّل رسالة.
  5. تمرين 5: استخدم اسم نمط (مثل #technology) ثم عدّل الرسالة.

قواعد PlantUML (قائمة كاملة)

  • skinparam يتحكم بالمحاذاة والخط والألوان.
  • يمكن وضع لون أو نمط بعد الدور.
  • الألوان يمكن أن تكون أسماء أو قيم Hex مثل #AliceBlue.
  • آخر تعريف لنفس الدور هو المعتمد.
  • الأنماط لا تغيّر صياغة الرسائل.

Sequence / CONTROL FLOW / Alt/Else (Choices)

هذا المثال يوضح “إذا… وإلا…”.

لماذا نستخدمه

  • عرض فروع مختلفة حسب الشروط.

كيفية الاستخدام

  1. استخدم alt للبداية، else لفرع آخر، و end للإنهاء.

استخدامات ممكنة

  • نجاح أو فشل الدفع.

تمارين (سهلة للأطفال)

  1. تمرين 1: غيّر الشروط إلى “مع مال / بلا مال” وعدّل الرسائل.
  2. تمرين 2: أضف فرع else ثالثًا وسطر رسالة.
  3. تمرين 3: غيّر عنوان alt إلى سيناريو من اختيارك وعدّل الرسائل.
  4. تمرين 4: أضف سطرًا في كل فرع وعدّل العناوين.
  5. تمرين 5: اجعل فرعًا “حاول مرة أخرى بعد الفشل” وفرعًا بنهاية مختلفة.

قواعد PlantUML (قائمة كاملة)

  • alt Condition تبدأ الفرع.
  • يمكن وجود عدة فروع else.
  • الإنهاء بـ end.
  • كل فرع يمكن أن يحتوي عدة رسائل.
  • نص الشرط اختياري لكن يُنصح به.

Sequence / CONTROL FLOW / Loop

هذا المثال يوضح التكرار.

لماذا نستخدمه

  • عرض خطوات متكررة.

كيفية الاستخدام

  1. استخدم loop و end لإحاطة التكرار.

استخدامات ممكنة

  • التحقق من الحالة كل ثانية.

تمارين (سهلة للأطفال)

  1. تمرين 1: غيّر عنوان الحلقة إلى “كل 3 مرات” وعدّل الرسالة.
  2. تمرين 2: أضف رسالة داخل الحلقة ثم اختصر الرسالة الأصلية.
  3. تمرين 3: غيّر أسماء الأدوار والرسالة إلى “جارٍ الفحص”.
  4. تمرين 4: غيّر عنوان الحلقة إلى شرط مثل “حتى الانتهاء” وعدّل الرسائل.
  5. تمرين 5: أضف رسائل بدء/انتهاء خارج الحلقة وخصّصها.

قواعد PlantUML (قائمة كاملة)

  • loop count/condition يمكن أن تكون في نفس السطر.
  • end تنهي كتلة الحلقة.
  • يمكن وضع عدة رسائل داخل الحلقة.

Sequence / CONTROL FLOW / Parallel (Par)

هذا المثال يوضح تنفيذ أمرين في وقت واحد.

لماذا نستخدمه

  • عرض المهام المتوازية.

كيفية الاستخدام

  1. استخدم par للبداية، else لفرع متوازٍ آخر، و end للإنهاء.

استخدامات ممكنة

  • إخطار خدمتين في الوقت نفسه.

تمارين (سهلة للأطفال)

  1. تمرين 1: غيّر الفروع إلى “إرسال بريد” و“إرسال SMS” وغيّر الأدوار.
  2. تمرين 2: غيّر الأدوار وأضف سطرًا في كل فرع.
  3. تمرين 3: أضف فرعًا ثالثًا (سطر else جديد) واكتب رسالة ثم اختصر أحد الأسطر.
  4. تمرين 4: غيّر أحد الفروع إلى “ببطء” وأضف رسالة نهاية وعدّل الفرع الآخر.
  5. تمرين 5: أضف رسالة “ابدأ المهمة” قبل par ورسالة “تم” بعد end.

قواعد PlantUML (قائمة كاملة)

  • par تنشئ كتلة متوازية.
  • لكل فرع تسلسل رسائل خاص به.
  • else ينتقل إلى فرع متوازٍ آخر.
  • الإنهاء بـ end.

Sequence / ADVANCED / Activation

هذا المثال يستخدم أشرطة التفعيل لإظهار انشغال الدور.

لماذا نستخدمه

  • عرض من يعمل ومدة العمل.

كيفية الاستخدام

  1. استخدم activate Role للبدء.
  2. استخدم deactivate Role للإنهاء.

استخدامات ممكنة

  • إظهار الخادم أثناء معالجة الطلب.

تمارين (سهلة للأطفال)

  1. تمرين 1: غيّر الدور إلى Server والرسالة إلى “جارٍ المعالجة”.
  2. تمرين 2: أضف سطر طلب ثم ضع activate بعده.
  3. تمرين 3: أضف رسالة “في المنتصف” ثم استبدلها بنصك.
  4. تمرين 4: أضف دورًا آخر بتفعيل ورسالة خاصة به.
  5. تمرين 5: حرّك بداية/نهاية التفعيل سطرًا واحدًا لأسفل وعدّل رسالة.

قواعد PlantUML (قائمة كاملة)

  • يمكن وضع activate/deactivate قبل أو بعد الرسائل.
  • activate و deactivate يجب أن يكونا متلازمين.
  • يمكن إدراج رسائل أثناء تفعيل الدور.

Sequence / ADVANCED / Notes

هذا المثال يوضح إضافة ملاحظات.

لماذا نستخدمه

  • إضافة شرح لزيادة الوضوح.

كيفية الاستخدام

  1. استخدم note left أو note right.
  2. اختم الملاحظة بـ end note.

استخدامات ممكنة

  • تذكير “قد تفشل هذه الخطوة”.

تمارين (سهلة للأطفال)

  1. تمرين 1: غيّر نص الملاحظة إلى تذكيرك.
  2. تمرين 2: انقل الملاحظة إلى اليمين وعدّل النص.
  3. تمرين 3: أضف ملاحظة ثانية قرب دور آخر.
  4. تمرين 4: اجعل الملاحظة سطرين وأضف سببًا في السطر الثاني.
  5. تمرين 5: انقل الملاحظة بعد رسالة أخرى.

قواعد PlantUML (قائمة كاملة)

  • يمكن وضع الملاحظات left أو right أو over.
  • الملاحظات متعددة الأسطر تستخدم note و end note.
  • الملاحظات لا تغيّر تدفق الرسائل.

Use Case / BASICS / Simple Use Case

مخططات حالات الاستخدام هي قائمة “ما الذي يمكن للنظام فعله”.

لماذا نستخدمه

  • توضّح وظائف المستخدم بوضوح.

كيفية الاستخدام

  1. استخدم actor للمستخدم.
  2. استخدم (UseCase) للوظيفة.

استخدامات ممكنة

  • تسجيل الدخول، البحث، الدفع… إلخ.

تمارين (سهلة للأطفال)

  1. تمرين 1: غيّر المستخدم إلى Teacher وغير الحالة إلى “إنشاء اختبار”.
  2. تمرين 2: أضف حالة استخدام ثانية.
  3. تمرين 3: غيّر أسماء الحالة والفاعل.
  4. تمرين 4: أضف فاعلًا ثانيًا ووصلْه بإحدى الحالات.
  5. تمرين 5: استخدم سيناريوك الخاص.

قواعد PlantUML (قائمة كاملة)

  • حالة الاستخدام تُكتب بين قوسين: (Login).
  • الفاعل يُكتب كشخص عصا: actor User.
  • صِل الفاعل بحالة الاستخدام بخط.

Use Case / RELATIONSHIPS / Include/Extend

يوضح الاحتواء أو الامتداد بين حالات الاستخدام.

لماذا نستخدمه

  • تمييز الخطوات المشتركة أو الاختيارية.

كيفية الاستخدام

  1. استخدم <> لـ “يشمل دائمًا”.
  2. استخدم <> لـ “اختياري”.

استخدامات ممكنة

  • تسجيل الدخول يشمل التحقق دائمًا؛ والدفع قد يمتد لخصم.

تمارين (سهلة للأطفال)

  1. تمرين 1: غيّر أسماء الحالات إلى مثالك.
  2. تمرين 2: بدّل بين include و extend.
  3. تمرين 3: أضف حالة استخدام أخرى وضمّنها.
  4. تمرين 4: اعكس السهم وعدّل النص.
  5. تمرين 5: غيّر نص العلاقة إلى صياغتك.

قواعد PlantUML (قائمة كاملة)

  • صيغة include: (A) .> (B) : <>.
  • صيغة extend: (A) .> (B) : <>.
  • استخدم سهامًا متقطعة.

Class / BASICS / Simple Class

مخططات الأصناف تصف السمات والأساليب.

لماذا نستخدمه

  • تنظيم هياكل البيانات.

كيفية الاستخدام

  1. استخدم class لتعريف صنف.
  2. ضع السمات والأساليب داخل الأقواس المعقوفة.

استخدامات ممكنة

  • تصميم نموذج بيانات البرنامج.

تمارين (سهلة للأطفال)

  1. تمرين 1: غيّر اسم الصنف إلى شيء تحبه مثل Book.
  2. تمرين 2: أضف سمة أو أسلوبًا واحدًا.
  3. تمرين 3: اختصر اسم الحقل أو ترجمه.
  4. تمرين 4: أضف صنفًا ثانيًا مع سمتين.
  5. تمرين 5: غيّر أسلوبًا إلى فعل من اختيارك.

قواعد PlantUML (قائمة كاملة)

  • صيغة الصنف: class Name { ... }.
  • السمات حقول، والأساليب تستخدم ().
  • يمكن ترك الصنف فارغًا.

Class / BASICS / Methods

هذا المثال يضيف أساليب وأنواع الإرجاع.

لماذا نستخدمه

  • وصف ما يمكن للصنف فعله.

كيفية الاستخدام

  1. تُكتب الأساليب كالتالي: name(): type.

استخدامات ممكنة

  • وصف واجهات API أو منطق الأعمال.

تمارين (سهلة للأطفال)

  1. تمرين 1: أضف أسلوبًا مع نوع إرجاع.
  2. تمرين 2: غيّر اسم أسلوب إلى فعل من اختيارك.
  3. تمرين 3: غيّر نوع الإرجاع إلى bool أو string.
  4. تمرين 4: أضف أسلوبًا آخر واجعل أحدهما خاصًا.
  5. تمرين 5: استخدم سيناريوك الخاص.

Class / VISIBILITY / Access Levels

هذا المثال يوضح الوصول العام والخاص والمحمي.

لماذا نستخدمه

  • فصل ما يمكن للآخرين الوصول إليه.

  1. + عام، - خاص، # محمي.

استخدامات ممكنة

  • إخفاء التفاصيل الداخلية.

تمارين (سهلة للأطفال)

  1. تمرين 1: اجعل حقلًا واحدًا خاصًا.
  2. تمرين 2: أضف حقلًا محميًا.
  3. تمرين 3: اجعل أسلوبًا عامًا.
  4. تمرين 4: أضف أسلوبًا خاصًا.
  5. تمرين 5: غيّر الأسماء إلى مثالك.

قواعد PlantUML (قائمة كاملة)

  • رموز الرؤية تسبق الحقول والأساليب.
  • الرموز الشائعة: + و - و # و ~.

Class / VISIBILITY / Abstract/Static

هذا المثال يوضح العلامات المجردة والثابتة.

لماذا نستخدمه

استخدامات ممكنة

  • أصناف أساسية وأساليب مساعدة.

تمارين (سهلة للأطفال)

  1. تمرين 1: اجعل أحد الأساليب ثابتًا.
  2. تمرين 2: غيّر اسم الصنف إلى مثال من اختيارك.
  3. تمرين 3: أضف أسلوبًا مجردًا.
  4. تمرين 4: استخدم أسلوب العلامة المجردة الأخرى.
  5. تمرين 5: استخدم سيناريوك الخاص.

قواعد PlantUML (قائمة كاملة)

  • {abstract} تحدد صنفًا أو أسلوبًا مجردًا.
  • {static} تحدد أسلوبًا ثابتًا.
