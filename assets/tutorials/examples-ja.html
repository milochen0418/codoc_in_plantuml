Sequence / BASICS / Simple Message

この例は「人Aが人Bへメッセージを送る」を示します。メモを渡すような感じです。

なぜ使うの？

  • 会話の順番が分かりやすくなります。
  • 「誰が最初に話し、次に誰が返事するか」を表すのに最適です。

使い方

  1. @startuml と @enduml を書きます。
  2. Alice -> Bob: Message で A が B に話していることを示します。

よくある用途

  • ログインの流れ：ユーザー → システム → サーバー。
  • チャットボット：あなたが質問 → ボットが回答。

練習（子ども向け）

  1. 練習1：Alice と Bob を好きな名前に変え、日常会話にします。
  2. 練習2：最初に participant 行を追加し、「先に質問、後で回答」に変えます。
  3. 練習3：メッセージを2行にし、片方を短くします。
  4. 練習4：誰が先に話すかを入れ替え、メッセージも変更します。
  5. 練習5：3人目を追加して一言言わせ、誰かの名前を変えます。

PlantUML 構文ルール（完全リスト）

  • @startuml で始め、@enduml で終わります。
  • 参加者は participant で宣言しても、直接使ってもOKです。
  • A -> B は呼び出し、A --> B は戻りです。
  • テキストはコロンの後：A -> B: Hello。
  • 上から下に時間が進みます。
  • 複数のメッセージで長い会話を表せます。
  • メッセージの文字は省略可能：A -> B でも有効です。
  • スペースを含む名前は引用符で囲みます。

Sequence / BASICS / Return Message

この例は「リクエストとレスポンス（質問と答え）」を示します。

なぜ使うの？

  • 要求と返答の手順がはっきりします。

使い方

  1. A -> B がリクエストです。
  2. B --> A がレスポンスです。

よくある用途

  • API 呼び出し：フロントエンド → バックエンド → データ返却。

練習（子ども向け）

  1. 練習1：役割を Client と Server にし、「データ取得 / データ返却」にします。
  2. 練習2：リクエストを「query」、返答を「query result」にします。
  3. 練習3：もう一組追加して（合計4行）2つの質問を表し、2つ目の文を変えます。
  4. 練習4：別の人が返答するようにして文を調整します。
  5. 練習5：リクエスト前に「準備する」の自己メッセージを追加します。

PlantUML 構文ルール（完全リスト）

  • -> はメッセージ送信。
  • --> は戻りを示すことが多い。
  • 返答の内容説明を追加できます。
  • 要求と返答はセットが多いですが必須ではありません。
  • 見やすさのために先に参加者を宣言できます。
  • 時間は上から下へ流れます。

Sequence / BASICS / Self-Message

「自分に話しかける」＝内部の処理を表します。

なぜ使うの？

  • オブジェクトが内部で作業していることを示せます。

使い方

  1. Alice -> Alice: Process

よくある用途

  • アルゴリズムの手順や内部計算。

練習（子ども向け）

  1. 練習1：Alice を自分の名前（例：Me）に変え、今日したいことにします。
  2. 練習2：「データ読み取り」に変え、Me -> Me: もう一度チェック を追加します。
  3. 練習3：「考える」に変え、Me -> Me: 重要点を書く を追加します。
  4. 練習4：名前を Robot にして Robot -> Robot: 充電 を追加します。
  5. 練習5：上に Me -> Me: start、下に Me -> Me: done を追加します。

PlantUML 構文ルール（完全リスト）

  • 矢印の両端が同じ参加者でもOKです。
  • 基本形式：Role -> Role: Text。
  • 名前は宣言なしでも使えます。
  • コロン後のテキストは省略可能。
  • 複数の自己メッセージで内部手順を表せます。
  • activate / deactivate と併用できます。
  • 時間は上から下へ流れます。

Sequence / PARTICIPANTS / Types

人・システム・データベースなど、役割によってアイコンが変わる例です。

なぜ使うの？

  • アイコンで役割がすぐ分かります。

使い方

  1. actor, boundary, control, database, entity を使います。

よくある用途

  • ログインの流れ：User → Web → Logic → Database。

練習（子ども向け）

  1. 練習1：User を Student に変え、メッセージを “login homework” にします。
  2. 練習2：boundary App を追加し、矢印をまず App に向けます。
  3. 練習3：database DB を database Books に変え、メッセージを変えます。
  4. 練習4：control 役割を追加し、1本やりとりを増やします。
  5. 練習5：参加者の順番を並べ替え、メッセージを調整します。

PlantUML 構文ルール（完全リスト）

  • 役割タイプは見た目だけを変え、名前は変えません。
  • やりとりには -> を使います。
  • 宣言形式：actor/boundary/control/database/entity Name。
  • 先に宣言してから名前を使えます。
  • 未宣言の名前は通常の参加者になります。

Sequence / PARTICIPANTS / Colors & Styles

色やスタイルで見やすくする例です。

なぜ使うの？

  • 重要な役割を色で目立たせられます。

使い方

  1. skinparam でスタイルを調整します。
  2. 役割の後に色を追加：participant Alice #AliceBlue。

よくある用途

  • 「中核システム」を明るい色で表示。

練習（子ども向け）

  1. 練習1：Alice を別名にして色を変えます。
  2. 練習2：メッセージを “start” にし、もう一方を暗い色にします。
  3. 練習3：3人目を追加して別の色にします。
  4. 練習4：skinparam sequenceMessageAlign を変えて一行を調整します。
  5. 練習5：スタイル名（例：#technology）を使い、メッセージを変えます。

PlantUML 構文ルール（完全リスト）

  • skinparam は配置・フォント・色などを制御します。
  • 役割には色やスタイル名を付けられます。
  • 色は名前や16進で指定できます（例：#AliceBlue）。
  • 同じ役割を再宣言した場合、最後のものが有効です。
  • スタイル設定はメッセージ構文を変えません。

Sequence / CONTROL FLOW / Alt/Else (Choices)

「もし…なら…そうでなければ…」を表します。

なぜ使うの？

  • 条件ごとの分岐を示せます。

使い方

  1. alt で開始、else で別分岐、end で終了。

よくある用途

  • 支払い成功／失敗。

練習（子ども向け）

  1. 練習1：「お金がある / ない」に変え、メッセージも変えます。
  2. 練習2：3つ目の else 分岐とメッセージを追加します。
  3. 練習3：alt のタイトルを自分のシナリオに変更します。
  4. 練習4：各分岐に1行追加し、タイトルも変えます。
  5. 練習5：片方を「失敗したら再試行」にします。

PlantUML 構文ルール（完全リスト）

  • alt Condition で分岐開始。
  • else は複数使えます。
  • end で終了。
  • 各分岐に複数メッセージを入れられます。
  • 条件テキストは任意ですが推奨です。

Sequence / CONTROL FLOW / Parallel (Par)

同時に進む処理を表します。

なぜ使うの？

  • 並行タスクを示せます。

使い方

  1. par で開始、else で並行分岐、end で終了。

よくある用途

  • 2つのサービスを同時に通知。

練習（子ども向け）

  1. 練習1：「メール送信」「SMS送信」に変え、役割名を変えます。
  2. 練習2：役割名を変え、各分岐に1行追加します。
  3. 練習3：3つ目の分岐（else）を追加します。
  4. 練習4：片方を「ゆっくり」にして終わりのメッセージを追加します。
  5. 練習5：par の前に「開始」、後に「完了」を追加します。

PlantUML 構文ルール（完全リスト）

  • par は並行ブロックを作ります。
  • 各分岐は独自のメッセージ順序を持てます。
  • else で別の並行分岐に切り替えます。
  • end で終了。

Sequence / ADVANCED / Activation

アクティベーションバーで「忙しい時間」を示します。

なぜ使うの？

  • 誰がどれだけ作業しているか分かります。

使い方

  1. activate Role で開始。
  2. deactivate Role で終了。

よくある用途

  • サーバーがリクエストを処理している様子。

練習（子ども向け）

  1. 練習1：役割を Server にし、メッセージを “processing” にします。
  2. 練習2：リクエスト行を追加し、その後に activate を置きます。
  3. 練習3：「halfway」を追加して自分の文に置き換えます。
  4. 練習4：別の役割を追加してアクティベーションを付けます。
  5. 練習5：アクティベーション開始/終了を1行下げます。

PlantUML 構文ルール（完全リスト）

  • activate/deactivate はメッセージの前後どちらでもOK。
  • activate と deactivate はペアで使います。
  • アクティブ中にメッセージを追加できます。

テーブル間の関係を示す例です。

なぜ使うの？

  • 1対多や多対多を示せます。

使い方

  1. User ||--o{ Post は 1対多 を意味します。

よくある用途

  • 1人のユーザーが多くの投稿を持つ。

練習（子ども向け）

  1. 練習1：User を Student、Post を Homework に変えてラベル更新。
  2. 練習2：1対多を1対1に変更し説明文も変更。
  3. 練習3：別のテーブルとの関係を追加して名前変更。
  4. 練習4：向きを入れ替えて名前を調整。
  5. 練習5：「1対多」のまま自分のシナリオを使う。

PlantUML 構文ルール（完全リスト）

  • || と o{ は多重度を示します。
  • ||--o{ は一般的に 1対多。
  • 線は2つのテーブルをつなぎます。
