Sequence / BASICS / Simple Message

Cet exemple montre “la personne A envoie un message à la personne B”, comme passer un mot.

Pourquoi l’utiliser

  • Montre l’ordre d’une conversation de façon claire.
  • Idéal pour les histoires “qui parle d’abord, qui répond ensuite”.

Comment l’utiliser

  1. Écris @startuml et @enduml.
  2. Utilise Alice -> Bob: Message pour montrer que A parle à B.

Cas d’usage possibles

  • Connexion : Utilisateur → Système → Serveur.
  • Chatbot : Tu demandes → Le bot répond.

Exercices (faciles pour les enfants)

  1. Exercice 1 : Change Alice et Bob par tes noms préférés et modifie le message.
  2. Exercice 2 : Ajoute une ligne participant puis change le message en “d’abord demander, ensuite répondre”.
  3. Exercice 3 : Mets le message sur deux lignes (question/réponse) et raccourcis l’une des lignes.
  4. Exercice 4 : Inverse qui parle en premier, puis modifie le texte.
  5. Exercice 5 : Ajoute un troisième personnage avec un message, puis renomme un personnage.

Règles de syntaxe PlantUML (liste complète)

  • Commence par @startuml et termine par @enduml.
  • Les participants peuvent être déclarés avec participant ou utilisés directement.
  • A -> B est un appel, A --> B est un retour.
  • Le texte vient après les deux-points : A -> B: Hello.
  • Le temps va de haut en bas.
  • Plusieurs messages montrent une conversation plus longue.
  • Le texte du message peut être omis : A -> B est valide.
  • Les noms avec espaces doivent être entre guillemets.

Sequence / BASICS / Return Message

Cet exemple montre “requête et réponse”, comme une question et une réponse.

Pourquoi l’utiliser

  • Affiche clairement les étapes de demande et de réponse.

Comment l’utiliser

  1. A -> B est la requête.
  2. B --> A est la réponse.

Cas d’usage possibles

  • Appel API : Frontend → Backend → Retour des données.

Exercices (faciles pour les enfants)

  1. Exercice 1 : Change les rôles en Client et Server, puis “obtenir des données / renvoyer des données”.
  2. Exercice 2 : Change la requête en “recherche” et la réponse en “résultat”.
  3. Exercice 3 : Ajoute une autre requête et réponse (4 lignes) et modifie le second texte.
  4. Exercice 4 : Fais répondre un autre personnage et ajuste le texte.
  5. Exercice 5 : Ajoute un message “préparation” avant la requête et rends-la plus précise.

Règles de syntaxe PlantUML (liste complète)

  • -> envoie un message.
  • --> indique souvent un retour.
  • Ajoute du texte pour décrire le contenu de la réponse.
  • Les requêtes et réponses vont souvent par paires, mais ce n’est pas obligatoire.
  • Déclare les participants si tu veux plus de clarté.
  • Le temps va de haut en bas.

Sequence / BASICS / Self-Message

C’est “se parler à soi-même”, pour montrer une étape interne.

Pourquoi l’utiliser

  • Montre qu’un objet fait un travail interne sans autres.

Comment l’utiliser

  1. Alice -> Alice: Process

Cas d’usage possibles

  • Étapes d’algorithme, calculs internes.

Exercices (faciles pour les enfants)

  1. Exercice 1 : Change Alice en ton nom (ex. Me) et modifie le message.
  2. Exercice 2 : Change l’action en “lire les données”, puis ajoute Me -> Me: vérifier encore.
  3. Exercice 3 : Change le message en “réfléchir”, puis ajoute Me -> Me: noter les points clés.
  4. Exercice 4 : Change le nom en Robot puis ajoute Robot -> Robot: recharger.
  5. Exercice 5 : Ajoute Me -> Me: start en haut et Me -> Me: done à la fin.

Règles de syntaxe PlantUML (liste complète)

  • Les deux extrémités de la flèche peuvent être le même participant.
  • Format de base : Role -> Role: Text.
  • Les noms peuvent être utilisés sans déclaration préalable.
  • Le texte après les deux-points est optionnel.
  • Plusieurs messages à soi-même indiquent plusieurs étapes internes.
  • Peut être combiné avec activate / deactivate.
  • Le temps va de haut en bas.

Comment l’utiliser

Exercices (faciles pour les enfants)

  • Les noms non déclarés sont des participants normaux.

Sequence / PARTICIPANTS / Colors & Styles

Cet exemple montre les couleurs et styles pour rendre le diagramme plus clair.

Pourquoi l’utiliser

  • Mettre en évidence les rôles importants.

Comment l’utiliser

  1. Utilise skinparam pour ajuster les styles.
  2. Ajoute une couleur après un rôle : participant Alice #AliceBlue.

Cas d’usage possibles

  • Afficher le “système central” en couleur vive.

Exercices (faciles pour les enfants)

  1. Exercice 1 : Renomme Alice et change la couleur.
  2. Exercice 2 : Change le message en “start” et assombrit l’autre rôle.
  3. Exercice 3 : Ajoute un troisième rôle avec une autre couleur.
  4. Exercice 4 : Change skinparam sequenceMessageAlign et ajuste un message.
  5. Exercice 5 : Utilise un nom de style (ex. #technology) puis change le message.

Règles de syntaxe PlantUML (liste complète)

  • skinparam contrôle l’alignement, la police, les couleurs, etc.
  • Les rôles peuvent être suivis d’une couleur ou d’un style.
  • Les couleurs peuvent être des noms ou des valeurs hex, comme #AliceBlue.
  • La dernière déclaration d’un même rôle est appliquée.
  • Les styles ne changent pas la syntaxe des messages.

Sequence / CONTROL FLOW / Alt/Else (Choices)

Cet exemple montre “si… sinon…”.

Pourquoi l’utiliser

  • Montrer différentes branches selon les conditions.

Comment l’utiliser

  1. Utilise alt pour commencer, else pour une autre branche, et end pour terminer.

Cas d’usage possibles

  • Paiement réussi ou échoué.

Exercices (faciles pour les enfants)

  1. Exercice 1 : Change les conditions en “avoir de l’argent / pas d’argent” et ajuste les messages.
  2. Exercice 2 : Ajoute une troisième branche else et un message.
  3. Exercice 3 : Change le titre de alt en ton scénario et modifie les messages.
  4. Exercice 4 : Ajoute une ligne dans chaque branche et change les titres.
  5. Exercice 5 : Fais une branche “réessayer après échec” et une autre avec une fin différente.

  • Il peut y avoir plusieurs branches else.
  • On termine par end.
  • Chaque branche peut contenir plusieurs messages.
  • Le texte de condition est optionnel mais recommandé.
