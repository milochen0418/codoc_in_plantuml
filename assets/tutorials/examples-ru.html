Sequence / BASICS / Simple Message

Этот пример показывает «Человек A отправляет одно сообщение человеку B», как передать записку.

Зачем использовать

  • Показывает порядок разговора, чтобы было легко понять.
  • Отлично подходит для историй «кто говорит первым, кто отвечает следующим».

Как использовать

  1. Напишите @startuml и @enduml.
  2. Используйте Alice -> Bob: Message, чтобы показать, что A говорит с B.

Возможные применения

  • Поток входа: Пользователь → Система → Сервер.
  • Чат‑бот: Вы спрашиваете → Бот отвечает.

Практика (простая для детей)

  1. Практика 1: Поменяйте Alice и Bob на любимые имена и сделайте сообщение про обычный разговор.
  2. Практика 2: Сначала добавьте строку participant, затем измените сообщение на «сначала спросить, потом ответить».
  3. Практика 3: Сделайте сообщение в две строки: сначала вопрос, затем ответ, и укоротите одну строку.
  4. Практика 4: Поменяйте, кто говорит первым, и измените текст сообщения.
  5. Практика 5: Добавьте третьего человека, который тоже скажет одну строку, и переименуйте одного участника.

Правила синтаксиса PlantUML (полный список)

  • Начинайте с @startuml и заканчивайте @enduml.
  • Участников можно объявлять через participant или использовать напрямую.
  • A -> B — вызов, A --> B — возврат.
  • Текст после двоеточия: A -> B: Hello.
  • Сверху вниз показывается порядок времени.
  • Несколько строк сообщений показывают более длинный разговор.
  • Текст сообщения можно опустить: A -> B корректно.
  • Имена с пробелами должны быть в кавычках.

Sequence / BASICS / Return Message

Этот пример показывает «запрос и ответ», как вопрос и ответ.

Зачем использовать

  • Четко показывает шаги запроса и ответа.

Как использовать

  1. A -> B — это запрос.
  2. B --> A — это ответ.

Возможные применения

  • Вызов API: Frontend → Backend → Возврат данных.

Практика (простая для детей)

  1. Практика 1: Поменяйте роли на Client и Server, а сообщения на «получить данные / вернуть данные».
  2. Практика 2: Поменяйте запрос на «query», а ответ на «query result».
  3. Практика 3: Добавьте еще один запрос и ответ (итого 4 строки) и измените второй текст.
  4. Практика 4: Пусть отвечает другой участник, и поправьте текст.
  5. Практика 5: Добавьте «подготовку» как self‑message перед запросом.

Правила синтаксиса PlantUML (полный список)

  • -> отправляет сообщение.
  • --> обычно означает возврат.
  • Можно добавить текст, описывающий ответ.
  • Запросы и ответы часто идут парой, но не обязательно.
  • Для ясности можно заранее объявить участников.
  • Время идет сверху вниз.

Sequence / BASICS / Self-Message

Это «разговор с самим собой», показывающий внутренний шаг.

Зачем использовать

  • Показывает, что объект делает внутреннюю работу.

Как использовать

  1. Alice -> Alice: Process

Возможные применения

  • Шаги алгоритма, внутренние вычисления.

Практика (простая для детей)

  1. Практика 1: Поменяйте Alice на свое имя (например, Me) и измените сообщение.
  2. Практика 2: Измените действие на «читать данные» и добавьте Me -> Me: проверить еще раз.
  3. Практика 3: Поменяйте сообщение на «думать» и добавьте Me -> Me: записать главное.
  4. Практика 4: Поменяйте имя на Robot и добавьте Robot -> Robot: зарядка.
  5. Практика 5: Добавьте сверху Me -> Me: start и снизу Me -> Me: done.

Правила синтаксиса PlantUML (полный список)

  • Оба конца стрелки могут быть у одного участника.
  • Базовый формат: Role -> Role: Text.
  • Имена можно использовать без предварительного объявления.
  • Текст после двоеточия необязателен.
  • Несколько self‑сообщений показывают несколько внутренних шагов.
  • Можно сочетать с activate / deactivate.
  • Время идет сверху вниз.

Sequence / PARTICIPANTS / Types

Этот пример показывает разные иконки для разных ролей: человек, система, база данных.

Зачем использовать

  • Иконки помогают быстрее распознавать роли.

Как использовать

  1. Используйте actor, boundary, control, database, entity.

Возможные применения

  • Поток входа: User → Web → Logic → Database.

Практика (простая для детей)

  1. Практика 1: Переименуйте User в Student и измените сообщение.
  2. Практика 2: Добавьте boundary App и направьте стрелку сначала к App.
  3. Практика 3: Переименуйте database DB в database Books и измените сообщение.
  4. Практика 4: Добавьте роль control и одну линию взаимодействия.
  5. Практика 5: Поменяйте порядок участников и скорректируйте сообщения.

Правила синтаксиса PlantUML (полный список)

  • Тип роли меняет внешний вид, но не имя.
  • Используйте -> для взаимодействий.
  • Формат объявления: actor/boundary/control/database/entity Name.
  • Сначала объявите роли, потом используйте имена.
  • Необъявленные имена — обычные участники.

Sequence / PARTICIPANTS / Colors & Styles

Этот пример показывает цвета и стили, чтобы диаграмма была понятнее.

Зачем использовать

  • Выделяйте важные роли цветом.

Как использовать

  1. Используйте skinparam для настройки стилей.
  2. Добавляйте цвет после роли: participant Alice #AliceBlue.

Возможные применения

  • Показать «ядро системы» ярким цветом.

Практика (простая для детей)

  1. Практика 1: Переименуйте Alice и поменяйте цвет.
  2. Практика 2: Поменяйте сообщение на «start» и сделайте другую роль темнее.
  3. Практика 3: Добавьте третью роль и назначьте другой цвет.
  4. Практика 4: Измените skinparam sequenceMessageAlign и поправьте строку.
  5. Практика 5: Используйте имя стиля (например, #technology) и измените сообщение.

Правила синтаксиса PlantUML (полный список)

  • skinparam управляет выравниванием, шрифтами, цветами и т. д.
  • Роли могут иметь цвет или имя стиля.
  • Цвета могут быть именами или hex, например #AliceBlue.
  • Если роль объявить снова, действует последнее объявление.
  • Настройки стиля не меняют синтаксис сообщений.

Sequence / CONTROL FLOW / Alt/Else (Choices)

Показывает «если… иначе…» — варианты выбора.

Зачем использовать

  • Показать разные ветки для разных условий.

Как использовать

  1. Используйте alt для старта, else для другой ветки и end для завершения.

Возможные применения

  • Оплата успешна или неуспешна.

Практика (простая для детей)

  1. Практика 1: Измените условия на «есть деньги / нет денег» и поменяйте сообщения.
  2. Практика 2: Добавьте третью ветку else и сообщение.
  3. Практика 3: Измените заголовок alt на свой сценарий.
  4. Практика 4: Добавьте по строке в каждую ветку и измените заголовки.
  5. Практика 5: Сделайте одну ветку «повторить после ошибки».

Правила синтаксиса PlantUML (полный список)

  • alt Condition начинает ветку.
  • Возможны несколько веток else.
  • Завершайте end.
  • Каждая ветка может содержать несколько сообщений.
  • Текст условия необязателен, но рекомендуется.

Как использовать

  1. Используйте loop и end, чтобы обрамить повторяемые строки.

Возможные применения

  • Проверять статус каждую секунду.

Практика (простая для детей)

  3. Практика 3: Переименуйте роли и измените сообщение на «проверка».

Правила синтаксиса PlantUML (полный список)

Sequence / CONTROL FLOW / Parallel (Par)

Показывает две вещи, происходящие одновременно.

Зачем использовать

  • Показывать параллельные задачи.

Как использовать

  1. Используйте par для старта, else для другой параллельной ветки и end для завершения.

Возможные применения

  • Уведомить два сервиса одновременно.

Практика (простая для детей)

  1. Практика 1: Поменяйте ветки на «отправить email» и «отправить SMS» и переименуйте роли.
  2. Практика 2: Переименуйте роли и добавьте по строке в каждую ветку.
  3. Практика 3: Добавьте третью ветку (еще один else) и сообщение.
  4. Практика 4: Одну ветку сделайте «медленно», добавьте сообщение завершения.
  5. Практика 5: Добавьте «начать задачу» до par и «готово» после end.

Правила синтаксиса PlantUML (полный список)

  • par создает параллельный блок.
  • Каждая ветка имеет свою последовательность сообщений.
  • else переключает на другую параллельную ветку.
  • Завершайте end.

Sequence / ADVANCED / Activation

Показывает, когда роль занята, с помощью полос активации.

Зачем использовать

  • Показать, кто работает и как долго.

Как использовать

  1. Используйте activate Role для начала.
  2. Используйте deactivate Role для окончания.

Возможные применения

  • Показать, как сервер обрабатывает запрос.

Практика (простая для детей)

  1. Практика 1: Поменяйте роль на Server и сообщение на “processing”.
  2. Практика 2: Добавьте строку запроса и поставьте activate после нее.
  3. Практика 3: Добавьте сообщение «halfway» и замените его своим текстом.
  4. Практика 4: Добавьте другую роль с активацией и сообщением.
  5. Практика 5: Сдвиньте начало/конец активации на строку ниже.

Правила синтаксиса PlantUML (полный список)

  • activate/deactivate могут быть до или после сообщений.
  • activate и deactivate идут парой.
  • Сообщения можно вставлять, пока роль активна.

Этот пример показывает отношения между таблицами.

Зачем использовать

  1. User ||--o{ Post означает один‑ко‑многим.

  • У одного пользователя много постов.

Практика (простая для детей)

  1. Практика 1: Переименуйте User в Student и Post в Homework, затем обновите подпись.
  2. Практика 2: Измените один‑ко‑многим на один‑к‑одному и обновите описание.
  3. Практика 3: Добавьте вторую связь и переименуйте таблицу.
  4. Практика 4: Поменяйте направление и имена.
  5. Практика 5: Оставьте «один‑ко‑многим» и используйте свой сценарий.

Правила синтаксиса PlantUML (полный список)

  • || и o{ показывают кратность.
  • ||--o{ обычно означает один‑ко‑многим.
