Sequence / BASICS / Simple Message

यह उदाहरण दिखाता है “व्यक्ति A, व्यक्ति B को एक संदेश भेजता है”, जैसे नोट पास करना।

क्यों इस्तेमाल करें

  • बातचीत का क्रम साफ़ दिखता है।
  • “कौन पहले बोलता है, कौन बाद में जवाब देता है” जैसे उदाहरणों के लिए अच्छा।

कैसे इस्तेमाल करें

  1. @startuml और @enduml लिखें।
  2. Alice -> Bob: Message से दिखाएँ कि A, B से बात कर रहा है।

संभावित उपयोग

  • लॉगिन फ्लो: उपयोगकर्ता → सिस्टम → सर्वर।
  • चैटबॉट: आप पूछते हैं → बॉट जवाब देता है।

अभ्यास (बच्चों के लिए आसान)

  1. अभ्यास 1: Alice और Bob को अपने पसंदीदा नामों से बदलें और संदेश बदलें।
  2. अभ्यास 2: पहले participant लाइन जोड़ें, फिर संदेश को “पहले पूछो, फिर जवाब दो” करें।
  3. अभ्यास 3: संदेश को दो लाइनों में बदलें (प्रश्न/उत्तर) और एक लाइन छोटी करें।
  4. अभ्यास 4: कौन पहले बोलता है बदलें और संदेश टेक्स्ट बदलें।
  5. अभ्यास 5: एक तीसरा व्यक्ति जोड़ें और किसी एक का नाम बदलें।

PlantUML सिंटैक्स नियम (पूरी सूची)

  • @startuml से शुरू करें और @enduml पर खत्म करें।
  • पार्टिसिपेंट participant से घोषित या सीधे नाम से लिखे जा सकते हैं।
  • A -> B कॉल है, A --> B रिटर्न है।
  • टेक्स्ट कॉलन के बाद लिखें: A -> B: Hello.
  • समय ऊपर से नीचे चलता है।
  • कई संदेश लंबी बातचीत दिखाते हैं।
  • संदेश का टेक्स्ट हटाया जा सकता है: A -> B वैध है।
  • स्पेस वाले नामों को कोट्स में लिखें।

Sequence / BASICS / Return Message

यह उदाहरण “अनुरोध और उत्तर” दिखाता है, जैसे प्रश्न और जवाब।

क्यों इस्तेमाल करें

  • अनुरोध और उत्तर के चरण साफ़ दिखते हैं।

कैसे इस्तेमाल करें

  1. A -> B अनुरोध है।
  2. B --> A उत्तर है।

संभावित उपयोग

  • API कॉल: फ्रंटएंड → बैकएंड → डेटा लौटाता है।

अभ्यास (बच्चों के लिए आसान)

  1. अभ्यास 1: रोल्स को Client और Server करें, संदेश “डेटा लो / डेटा दो” करें।
  2. अभ्यास 2: अनुरोध “query” और उत्तर “result” करें।
  3. अभ्यास 3: एक और अनुरोध/उत्तर जोड़ें (कुल चार लाइनें) और दूसरे टेक्स्ट को बदलें।
  4. अभ्यास 4: किसी और से जवाब दिलवाएँ और टेक्स्ट समायोजित करें।
  5. अभ्यास 5: अनुरोध से पहले “तैयारी” वाला self-message जोड़ें।

PlantUML सिंटैक्स नियम (पूरी सूची)

  • -> संदेश भेजने के लिए।
  • --> अक्सर रिटर्न संदेश दिखाता है।
  • उत्तर के टेक्स्ट का वर्णन जोड़ सकते हैं।
  • अनुरोध/उत्तर आमतौर पर जोड़े में होते हैं, लेकिन जरूरी नहीं।
  • स्पष्टता के लिए पार्टिसिपेंट पहले घोषित कर सकते हैं।
  • समय ऊपर से नीचे चलता है।

Sequence / BASICS / Self-Message

यह “खुद से बात” दिखाता है, यानी अंदर का कदम।

क्यों इस्तेमाल करें

  • दिखाता है कि ऑब्जेक्ट अंदरूनी काम कर रहा है।

कैसे इस्तेमाल करें

  1. Alice -> Alice: Process

संभावित उपयोग

  • एल्गोरिदम के चरण, अंदरूनी गणना।

अभ्यास (बच्चों के लिए आसान)

  1. अभ्यास 1: Alice को अपने नाम से बदलें और संदेश बदलें।
  2. अभ्यास 2: एक्शन “डेटा पढ़ो” करें और Me -> Me: फिर जांचो जोड़ें।
  3. अभ्यास 3: संदेश “सोचो” करें और Me -> Me: मुख्य बातें लिखो जोड़ें।
  4. अभ्यास 4: नाम Robot करें और Robot -> Robot: चार्ज जोड़ें।
  5. अभ्यास 5: ऊपर Me -> Me: start और अंत में Me -> Me: done जोड़ें।

PlantUML सिंटैक्स नियम (पूरी सूची)

  • तीर के दोनों सिरे एक ही पार्टिसिपेंट हो सकते हैं।
  • बेसिक फ़ॉर्मेट: Role -> Role: Text.
  • नाम बिना घोषणा के उपयोग हो सकते हैं।
  • कॉलन के बाद टेक्स्ट वैकल्पिक है।
  • कई self-messages कई आंतरिक चरण दिखाते हैं।
  • activate / deactivate के साथ उपयोग कर सकते हैं।
  • समय ऊपर से नीचे चलता है।

Sequence / PARTICIPANTS / Types

यह उदाहरण अलग-अलग रोल्स के लिए अलग आइकॉन दिखाता है (व्यक्ति, सिस्टम, डेटाबेस)।

क्यों इस्तेमाल करें

  • आइकॉन से रोल्स जल्दी समझ आते हैं।

कैसे इस्तेमाल करें

  1. actor, boundary, control, database, entity इस्तेमाल करें।

संभावित उपयोग

  • लॉगिन फ्लो: User → Web → Logic → Database.

अभ्यास (बच्चों के लिए आसान)

  1. अभ्यास 1: User को Student करें और संदेश बदलें।
  2. अभ्यास 2: boundary App जोड़ें और तीर पहले App पर ले जाएँ।
  3. अभ्यास 3: database DB को database Books करें और संदेश बदलें।
  4. अभ्यास 4: एक control रोल और एक इंटरैक्शन लाइन जोड़ें।
  5. अभ्यास 5: रोल्स का क्रम बदलें और संदेश समायोजित करें।

PlantUML सिंटैक्स नियम (पूरी सूची)

  • रोल टाइप केवल रूप बदलता है, नाम नहीं।
  • इंटरैक्शन के लिए -> उपयोग करें।
  • डिक्लेरेशन फ़ॉर्मेट: actor/boundary/control/database/entity Name.
  • रोल्स पहले घोषित करें, फिर नाम उपयोग करें।
  • बिना घोषणा के नाम सामान्य पार्टिसिपेंट हैं।

Sequence / PARTICIPANTS / Colors & Styles

यह उदाहरण रंग और स्टाइल दिखाता है ताकि डायग्राम साफ़ दिखे।

क्यों इस्तेमाल करें

  • महत्वपूर्ण रोल्स को रंग से हाईलाइट करें।

कैसे इस्तेमाल करें

  1. स्टाइल बदलने के लिए skinparam उपयोग करें।
  2. रोल के बाद रंग जोड़ें: participant Alice #AliceBlue.

संभावित उपयोग

  • “मुख्य सिस्टम” को चमकीले रंग में दिखाना।

अभ्यास (बच्चों के लिए आसान)

  1. अभ्यास 1: Alice और रंग बदलें।
  2. अभ्यास 2: संदेश “start” करें और दूसरे रोल को गहरा रंग दें।
  3. अभ्यास 3: तीसरा रोल जोड़ें और अलग रंग दें।
  4. अभ्यास 4: skinparam sequenceMessageAlign बदलें और संदेश बदलें।
  5. अभ्यास 5: स्टाइल नाम (जैसे #technology) दें और संदेश बदलें।

PlantUML सिंटैक्स नियम (पूरी सूची)

  • skinparam अलाइनमेंट, फ़ॉन्ट, रंग आदि नियंत्रित करता है।
  • रोल के बाद रंग या स्टाइल नाम आ सकता है।
  • रंग नाम या हेक्स हो सकता है, जैसे #AliceBlue.
  • उसी रोल की आखिरी घोषणा मान्य होगी।
  • स्टाइल सेटिंग्स संदेश सिंटैक्स नहीं बदलतीं।

Sequence / CONTROL FLOW / Alt/Else (Choices)

यह उदाहरण “यदि… वरना…” दिखाता है।

क्यों इस्तेमाल करें

  • अलग-अलग स्थितियों के लिए शाखाएँ दिखाता है।

कैसे इस्तेमाल करें

  1. alt से शुरू करें, else से दूसरी शाखा, और end से खत्म करें।

संभावित उपयोग

  • पेमेंट सफल/विफल।

अभ्यास (बच्चों के लिए आसान)

  1. अभ्यास 1: शर्तें “पैसे हैं / पैसे नहीं” करें और संदेश बदलें।
  2. अभ्यास 2: तीसरी else शाखा जोड़ें और संदेश लिखें।
  3. अभ्यास 3: alt शीर्षक को अपने परिदृश्य से बदलें।
  4. अभ्यास 4: हर शाखा में एक लाइन जोड़ें और शीर्षक बदलें।
  5. अभ्यास 5: एक शाखा “फिर से कोशिश” करें और दूसरी का अंत बदलें।

PlantUML सिंटैक्स नियम (पूरी सूची)

  • alt Condition शाखा शुरू करता है।
  • कई else हो सकते हैं।
  • end से समाप्त करें।
  • हर शाखा में कई संदेश हो सकते हैं।
  • शर्त का टेक्स्ट वैकल्पिक है लेकिन सुझाया जाता है।

Sequence / CONTROL FLOW / Loop

यह उदाहरण दोहराव दिखाता है।

क्यों इस्तेमाल करें

  • बार-बार होने वाले कदम दिखते हैं।

कैसे इस्तेमाल करें

  1. दोहराव को loop और end से घेरें।

संभावित उपयोग

  • हर सेकंड स्थिति जाँचना।

अभ्यास (बच्चों के लिए आसान)

  1. अभ्यास 1: loop शीर्षक “हर 3 बार” करें और संदेश बदलें।
  2. अभ्यास 2: loop के अंदर एक संदेश जोड़ें और मूल संदेश छोटा करें।
  3. अभ्यास 3: रोल नाम बदलें और संदेश “चेकिंग” करें।
  4. अभ्यास 4: loop शीर्षक को “जब तक पूरा न हो” जैसी शर्त में बदलें।
  5. अभ्यास 5: loop के बाहर शुरुआत/समाप्ति संदेश जोड़ें।

PlantUML सिंटैक्स नियम (पूरी सूची)

  • loop count/condition एक ही लाइन में हो सकता है।
  • end loop ब्लॉक खत्म करता है।
  • loop के अंदर कई संदेश हो सकते हैं।

Sequence / CONTROL FLOW / Parallel (Par)

यह उदाहरण एक साथ दो काम दिखाता है।

क्यों इस्तेमाल करें

  • पैरेलल काम दिखाने के लिए।

कैसे इस्तेमाल करें

  1. par से शुरू करें, else से दूसरी शाखा, end से खत्म करें।

संभावित उपयोग

  • दो सेवाओं को साथ में नोटिफ़ाई करना।

अभ्यास (बच्चों के लिए आसान)

  1. अभ्यास 1: शाखाएँ “ईमेल भेजो” और “SMS भेजो” करें और रोल नाम बदलें।
  2. अभ्यास 2: रोल नाम बदलें और हर शाखा में एक लाइन जोड़ें।
  3. अभ्यास 3: तीसरी शाखा (else) जोड़ें और एक लाइन लिखें।
  4. अभ्यास 4: एक शाखा “धीरे करो” करें और दूसरी शाखा बदलें।
  5. अभ्यास 5: par से पहले “कार्य शुरू” और बाद में “समाप्त” जोड़ें।

PlantUML सिंटैक्स नियम (पूरी सूची)

  • par पैरेलल ब्लॉक बनाता है।
  • हर शाखा का अपना संदेश क्रम होता है।
  • else दूसरी पैरेलल शाखा पर जाता है।
  • end से समाप्त करें।

Sequence / ADVANCED / Activation

यह उदाहरण activation bars दिखाता है कि रोल कब व्यस्त है।

क्यों इस्तेमाल करें

  • कौन काम कर रहा है और कितनी देर तक।

कैसे इस्तेमाल करें

  1. activate Role से शुरू करें।
  2. deactivate Role से समाप्त करें।

संभावित उपयोग

  • सर्वर का अनुरोध प्रोसेस करना।

अभ्यास (बच्चों के लिए आसान)

  1. अभ्यास 1: रोल को Server करें और संदेश “processing” करें।
  2. अभ्यास 2: एक अनुरोध लाइन जोड़ें और उसके बाद activate रखें।
  3. अभ्यास 3: “halfway” संदेश जोड़ें और उसे अपने टेक्स्ट से बदलें।
  4. अभ्यास 4: एक और रोल जोड़ें और उसका activation दिखाएँ।
  5. अभ्यास 5: activation की शुरुआत/अंत एक लाइन नीचे करें।

PlantUML सिंटैक्स नियम (पूरी सूची)

  • activate/deactivate संदेशों से पहले या बाद में आ सकते हैं।
  • activate और deactivate जोड़ी में होते हैं।
  • activation के दौरान संदेश जोड़े जा सकते हैं।

Sequence / ADVANCED / Notes

यह उदाहरण नोट्स जोड़ना दिखाता है।

क्यों इस्तेमाल करें

  • अतिरिक्त समझाने के लिए।

कैसे इस्तेमाल करें

  1. note left या note right उपयोग करें।
  2. end note से नोट खत्म करें।

संभावित उपयोग

  • “यह चरण असफल हो सकता है” जैसी सूचना।

अभ्यास (बच्चों के लिए आसान)

  1. अभ्यास 1: नोट टेक्स्ट बदलें।
  2. अभ्यास 2: नोट को दाईं ओर ले जाएँ और टेक्स्ट बदलें।
  3. अभ्यास 3: दूसरा नोट जोड़ें।
  4. अभ्यास 4: नोट को दो लाइनों में करें।
  5. अभ्यास 5: नोट को किसी दूसरे संदेश के बाद रखें।

PlantUML सिंटैक्स नियम (पूरी सूची)

  • नोट्स left, right या over हो सकते हैं।
  • मल्टी-लाइन नोट्स में note और end note उपयोग करें।
  • नोट्स संदेश फ्लो नहीं बदलते।

Database (ER) / ENTITIES / Basic Entity

ER डायग्राम डेटाबेस तालिकाएँ दिखाते हैं।

क्यों इस्तेमाल करें

  • डेटाबेस संरचना डिजाइन करने के लिए।

कैसे इस्तेमाल करें

  1. entity से टेबल परिभाषित करें।
  2. Primary key के लिए * उपयोग करें।

संभावित उपयोग

  • यूज़र टेबल डिजाइन।

अभ्यास (बच्चों के लिए आसान)

  1. अभ्यास 1: टेबल को Book और फ़ील्ड्स को title, author करें।
  2. अभ्यास 2: एक field जोड़ें और एक type को int करें।
  3. अभ्यास 3: primary key का नाम बदलें और type बदलें।
  4. अभ्यास 4: दूसरी टेबल और दो फ़ील्ड्स जोड़ें।
  5. अभ्यास 5: separator line जोड़ें और फ़ील्ड्स का क्रम बदलें।

PlantUML सिंटैक्स नियम (पूरी सूची)

  • Separator line -- सेक्शन अलग करता है।
  • * primary key field दिखाता है।
  • Field फ़ॉर्मेट: name: type.

Database (ER) / ENTITIES / Relationships

यह उदाहरण तालिकाओं के बीच संबंध दिखाता है।

क्यों इस्तेमाल करें

  • One-to-many या many-to-many दिखाने के लिए।

कैसे इस्तेमाल करें

  1. User ||--o{ Post का अर्थ one-to-many है।

संभावित उपयोग

  • एक user की कई posts।

अभ्यास (बच्चों के लिए आसान)

  1. अभ्यास 1: User को Student और Post को Homework करें।
  2. अभ्यास 2: one-to-many को one-to-one करें और विवरण बदलें।
  3. अभ्यास 3: दूसरी relation जोड़ें और नाम बदलें।
  4. अभ्यास 4: दिशा बदलें और नाम समायोजित करें।
  5. अभ्यास 5: “one-to-many” रखें और अपना परिदृश्य उपयोग करें।

PlantUML सिंटैक्स नियम (पूरी सूची)

  • || और o{ cardinality दिखाते हैं।
  • ||--o{ अक्सर one-to-many होता है।
  • लाइन दो तालिकाओं को जोड़ती है।
