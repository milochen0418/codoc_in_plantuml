Sequence / BASICS / Simple Message

这个范例告诉你“角色 A 对角色 B 说一句话”。就像两个人互相传纸条。

为什么要用

  • 把系统里的对话顺序画出来，老师或同学一看就懂。
  • 适合讲“谁先说、谁再回”的故事。

怎么用

  1. 先写 @startuml 和 @enduml。
  2. 用 Alice -> Bob: 消息 表示 A 对 B 说话。

可能的用法

  • 登录流程：用户 → 系统 → 服务器。
  • 聊天机器人：你问 → 机器人回。

修改练习（12 岁也懂）

  1. 练习一：把 Alice、Bob 改成你喜欢的角色名字，再把消息改成一句日常对话。
  2. 练习二：先新增一行 participant 声明角色，再把原本的消息改成“先问、再回”的内容。
  3. 练习三：把消息改成两行，第一行问问题，第二行请对方回答，再把其中一行换成更短的文字。
  4. 练习四：把对话顺序换一下（谁先说换成另一个人），再改消息内容。
  5. 练习五：加一个第三个角色，让他也说一句话，再把其中一个角色改名。

PlantUML 语法规则（完整清单）

  • 以 @startuml 开头、@enduml 结尾。
  • 角色可以写 participant 或直接用名字。
  • A -> B 是同步消息，A --> B 是回传。
  • 文字写在冒号后面：A -> B: Hello。
  • 顺序由上到下表示时间。
  • 可以连续多行消息，表示多段对话。
  • 消息文字可省略：A -> B 也合法。
  • 角色名称可包含空格，需用引号包起来。

Sequence / BASICS / Return Message

这个范例示范“请求与回应”。像你问问题，对方回答。

为什么要用

  • 清楚表达“发问”和“回复”两个步骤。

怎么用

  1. A -> B 表示请求。
  2. B --> A 表示回应。

可能的用法

  • API 调用：前端 → 后端 → 回传资料。

修改练习（12 岁也懂）

  1. 练习一：把角色改成 Client 与 Server，再把消息改成“要资料 / 传资料”。
  2. 练习二：先把请求消息改成“查询”，再把回复消息改成“查询结果”。
  3. 练习三：加一个新的请求和回复（共四行），表示连续问两次，再把第二次的文字改成不同内容。
  4. 练习四：把回复改成另一个人回，并调整消息文字。
  5. 练习五：在请求前加一行“准备中”的自我消息，再把请求文字改得更明确。

PlantUML 语法规则（完整清单）

  • -> 用于发出消息。
  • --> 常用于回传消息。
  • 可加上文字描述回传内容。
  • 请求与回应通常成对出现，但不是硬性规定。
  • 可在消息前先声明角色，让名称更清楚。
  • 消息顺序由上到下代表时间先后。

Sequence / BASICS / Self-Message

这是“自己跟自己说话”，代表自己在做内部步骤。

为什么要用

  • 显示对象自己进行处理，不需要别人帮忙。

怎么用

  1. Alice -> Alice: Process

可能的用法

  • 算法步骤、内部计算。

修改练习（12 岁也懂）

  1. 练习一：先把 Alice 改成你喜欢的名字（例如 Me），再把消息改成你今天想做的事。
  2. 练习二：先把第一行的动作改成 先读资料，再加一行 Me -> Me: 再检查。
  3. 练习三：先把消息改成 想一想，再加一行 Me -> Me: 记下重点。
  4. 练习四：先把名字改成 Robot，再加一行 Robot -> Robot: 充电。
  5. 练习五：先新增一行 Me -> Me: 开始 放在最上面，再在最后加一行 Me -> Me: 完成。

PlantUML 语法规则（完整清单）

  • 箭头左右两边可以是同一个角色。
  • 基本格式：角色 -> 角色: 文字。
  • 角色名字可直接使用，不一定要先声明。
  • 消息文字写在冒号后面，没有文字也可以。
  • 同一个角色可连续写多行，表示多个内部步骤。
  • 可搭配 activate / deactivate 显示忙碌时间。
  • 顺序由上到下表示时间先后。

Sequence / PARTICIPANTS / Types

这个范例让不同角色用不同图示出现，比如“人、系统、数据库”。

为什么要用

  • 用图示区分角色，读图更快。

怎么用

  1. 用 actor、boundary、control、database、entity。

可能的用法

  • 登录流程：用户 → 网页 → 逻辑 → 数据库。

修改练习（12 岁也懂）

  1. 练习一：把 User 改成 Student，再把消息改成“登录作业”。
  2. 练习二：先加一个 boundary App，再把箭头改成先到 App。
  3. 练习三：把 database DB 改成 database Books，再改消息内容。
  4. 练习四：新增一个 control 角色，并加一条互动消息。
  5. 练习五：把角色顺序调整，再让消息也跟着改顺序。

PlantUML 语法规则（完整清单）

  • 角色类型会改变外观但名字不变。
  • 同样可以使用 -> 画互动。
  • 角色声明格式：actor/ boundary/ control/ database/ entity 名称。
  • 角色可以先声明，再在消息中直接使用名字。
  • 未声明的名字会被当作一般角色。

Sequence / PARTICIPANTS / Colors & Styles

这个范例示范“上色与样式”，让图更漂亮、重点更清楚。

为什么要用

  • 用颜色标示重要角色。

怎么用

  1. 用 skinparam 调整样式。
  2. 在角色后面加颜色：participant Alice #AliceBlue。

可能的用法

  • 把“核心系统”用亮色表示。

修改练习（12 岁也懂）

  1. 练习一：把 Alice 改成你喜欢的名字，再换成另一种颜色。
  2. 练习二：先把消息改成“开始”，再把另一个角色改成深色。
  3. 练习三：新增第三个角色，并替它指定不同颜色。
  4. 练习四：调整 skinparam sequenceMessageAlign 的值，再换一条消息。
  5. 练习五：把一位角色换成样式名称（如 #technology），再改消息内容。

PlantUML 语法规则（完整清单）

  • skinparam 可调整对齐、字体、颜色等。
  • 角色后可接颜色或样式名称。
  • 颜色可用色名或色码，例如 #AliceBlue。
  • 同一角色若重复声明，以最后一次为准。
  • 样式设置不影响消息语法。

Sequence / CONTROL FLOW / Alt/Else (Choices)

这个范例表示“如果…不然…”，像选择题。

为什么要用

  • 画出不同情况的分支行为。

怎么用

  1. 用 alt 开始，else 切换分支，end 结束。

可能的用法

  • 付款成功或失败的流程。

修改练习（12 岁也懂）

  1. 练习一：把条件改成“有钱 / 没钱”，再改两边的消息文字。
  2. 练习二：先新增第三个 else 分支，再补一行消息。
  3. 练习三：把 alt 的标题改成你自己的情境，再改消息内容。
  4. 练习四：在每个分支各加一行消息，让步骤更清楚，再把分支标题改成你自己的情境。
  5. 练习五：把其中一个分支改成“失败后再试一次”的对话，再把另一个分支改成不同结局。

PlantUML 语法规则（完整清单）

  • alt 条件 开始分支。
  • 可有多个 else。
  • 最后以 end 结束。
  • 每个分支内都可以放多行消息。
  • 条件文字可省略，但建议写清楚。

Sequence / CONTROL FLOW / Loop

这个范例表示“重复做事情”。

为什么要用

  • 展示多次重复的动作。

怎么用

  1. 用 loop 与 end 包住重复内容。

可能的用法

  • 每秒检查一次状态。

修改练习（12 岁也懂）

  1. 练习一：把 loop 的文字改成“每 3 次”，再改消息内容。
  2. 练习二：在循环内加一行新消息，表示两个动作，再把原本的消息改短。
  3. 练习三：把角色名称改掉，再把消息改成“检查中”。
  4. 练习四：把循环标题改成条件式，例如“只要还没完成”，再把消息内容改掉。
  5. 练习五：在循环外加一行开始与结束的消息，再把开始或结束改成你自己的句子。

PlantUML 语法规则（完整清单）

  • loop 次数/条件 可写在同一行。
  • end 结束循环区块。
  • 循环内可有多行消息。

Sequence / CONTROL FLOW / Parallel (Par)

这个范例表示“同时做两件事”。

为什么要用

  • 显示并行工作。

怎么用

  1. 用 par 开始，else 切换并行分支，end 结束。

可能的用法

  • 同时通知两个服务。

修改练习（12 岁也懂）

  1. 练习一：把两个分支的消息改成“发信”和“发短信”，再把角色名字改成你喜欢的。
  2. 练习二：先把角色改名，再把每个分支多加一行消息。
  3. 练习三：新增第三个并行分支（再加一个 else），并写一行消息，再把其中一行改短。
  4. 练习四：把其中一个分支改成“慢慢做”，并加一行结束消息，再改另一分支文字。
  5. 练习五：先在 par 前加一行“开始任务”，再在 end 后加一行“完成”。

PlantUML 语法规则（完整清单）

  • par 代表并行区块。
  • 每个分支可有自己的消息序列。
  • else 用来切换到另一个并行分支。
  • 最后以 end 结束并行区块。

Sequence / ADVANCED / Activation

这个范例用“激活条”表示某个角色正在忙碌。

为什么要用

  • 表达谁在执行工作，以及工作持续多久。

怎么用

  1. 用 activate 角色 开始。
  2. 用 deactivate 角色 结束。

可能的用法

  • 显示系统正在处理请求。

修改练习（12 岁也懂）

  1. 练习一：把角色改成 Server，再把消息改成“处理中”。
  2. 练习二：先加一条请求消息，再把 activate 放在请求后。
  3. 练习三：在忙碌期间加一行消息，表示“完成一半”，再把这行改成你自己的文字。
  4. 练习四：新增另一个角色，让它也有一段忙碌时间，再加一条它的消息。
  5. 练习五：把忙碌开始与结束的位置往下移一行，观察差异，再调整其中一条消息。

PlantUML 语法规则（完整清单）

  • 可在消息前后加入 activate/deactivate。
  • activate 角色 与 deactivate 角色 要成对。
  • 可在忙碌期间插入多行消息。

这个范例示范在图上加小纸条说明。

为什么要用

  • 补充解释，让看的人更清楚。

怎么用

  1. 用 note left 或 note right。
  2. 用 end note 结束注解。

可能的用法

  • 提醒“此步骤可能失败”。

修改练习（12 岁也懂）

  1. 练习一：把注解文字改成你自己的提醒句。
  2. 练习二：把注解位置改成 note right，再改文字。
